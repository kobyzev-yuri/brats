# Резюме: Созданные скрипты для sales-agent

## Дата создания
2026-02-08

## Созданные скрипты

### ✅ 1. `extract_kb_from_conversations.py` - Извлечение знаний из диалогов

**Назначение:** Автоматическое извлечение полезной информации из успешных диалогов для обогащения KB.

**Функциональность:**
- Анализ успешных диалогов (привели к КП/сделке)
- Извлечение эффективных ответов на вопросы
- Извлечение успешных паттернов обработки возражений
- Улучшение скриптов продаж по этапам FSM
- Автоматическое добавление в KB с улучшением через LLM

**Использование:**
```bash
python scripts/extract_kb_from_conversations.py --days 30 --min-success-rate 0.7 --dry-run
```

**Связь с архитектурой:**
- Использует FSM состояния из `BUSINESS_PROCESSES.md`
- Интегрируется с KB Service для обогащения базы знаний
- Следует принципам модульности и переиспользования сервисов

---

### ✅ 2. `test_conversation_flow.py` - Тестирование FSM воронки

**Назначение:** Автоматическое тестирование состояний FSM и переходов между ними.

**Функциональность:**
- Тестирование всех состояний FSM (GREETING, QUALIFYING, PROPOSAL и т.д.)
- Проверка валидности переходов между состояниями
- Валидация схемы БД для поддержки FSM
- Проверка значений состояний в реальных диалогах
- Выявление недопустимых переходов и застрявших диалогов

**Использование:**
```bash
python scripts/test_conversation_flow.py --test-all
python scripts/test_conversation_flow.py --test-state QUALIFYING --verbose
```

**Связь с архитектурой:**
- Основан на FSM диаграмме из `BUSINESS_PROCESSES.md`
- Использует валидные переходы согласно архитектуре
- Проверяет соответствие БД схеме из `db-init/03_create_conversations.sql`

---

### ✅ 3. `simulate_conversation.py` - Симуляция диалога

**Назначение:** Симуляция диалога для тестирования без реального клиента.

**Функциональность:**
- Создание тестового диалога в БД
- Отправка сообщений от имени "клиента"
- Проверка ответов агента (когда API будет реализован)
- Валидация переходов состояний FSM
- Поддержка предопределенных сценариев и интерактивного режима

**Использование:**
```bash
python scripts/simulate_conversation.py --scenario objection_handling
python scripts/simulate_conversation.py --interactive
```

**Доступные сценарии:**
- `objection_handling` - обработка возражений
- `qualification` - квалификация лида
- `proposal_generation` - генерация КП
- `handoff` - передача менеджеру

**Связь с архитектурой:**
- Тестирует FSM состояния и переходы из `BUSINESS_PROCESSES.md`
- Симулирует реальные сценарии из воронки продаж

---

### ✅ 4. `analyze_conversations.py` - Анализ эффективности диалогов

**Назначение:** Анализ завершенных диалогов для выявления паттернов успеха/неудачи.

**Функциональность:**
- Статистика по диалогам (количество, средняя длительность, конверсия)
- Определение успешных диалогов (привели к КП/сделке)
- Выявление паттернов успешных сообщений
- Анализ причин handoff
- Генерация рекомендаций по улучшению

**Использование:**
```bash
python scripts/analyze_conversations.py --days 30
python scripts/analyze_conversations.py --days 30 --output report.json --format json
```

**Выходные данные:**
- Статистика по диалогам и состояниям FSM
- Топ успешных фраз/паттернов
- Причины handoff (частота, категории)
- Рекомендации по улучшению

**Связь с архитектурой:**
- Анализирует диалоги согласно процессам из `BUSINESS_PROCESSES.md`
- Использует состояния FSM для категоризации
- Интегрируется с таблицами `conversations` и `proposals`

## Принципы реализации

### 1. Соответствие архитектуре

Все скрипты созданы на основе:
- **BUSINESS_PROCESSES.md** - FSM состояния, переходы, процессы
- **db-init/03_create_conversations.sql** - схема таблиц
- **docs/CONFIGURATION_REFERENCE.md** - конфигурация

### 2. Использование сервисов

- Все скрипты используют `EmbeddingService` и `LLMService` из `kb-service`
- Модели читаются из `config.env`, а не хардкодятся
- Переиспользование кода между скриптами

### 3. Поддержка dry-run режима

- Все скрипты поддерживают `--dry-run` для тестирования без изменений
- Позволяет проверить, что будет сделано, перед реальным выполнением

### 4. Обработка ошибок

- Graceful degradation при недоступности сервисов
- Подробное логирование ошибок
- Продолжение работы при ошибках в отдельных модулях

### 5. Модульность

- Каждый скрипт может работать независимо
- Легко расширять и добавлять новые функции
- Переиспользование общих компонентов

---

## Конфигурация

Все скрипты используют переменные из `config.env`:

**Обязательные:**
- `DATABASE_URL` - подключение к PostgreSQL

**Опциональные:**
- `KB_API_URL` - URL KB Service API (по умолчанию `http://localhost:8001`)
- `SALES_AGENT_URL` - URL Sales Agent API (когда будет реализован)
- `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL` - для LLM
- `EMBEDDING_PROVIDER`, `EMBEDDING_MODEL` или `HF_MODEL_NAME` - для embeddings

**См. также:**
- [CONFIGURATION_REFERENCE.md](../../docs/CONFIGURATION_REFERENCE.md) - полный справочник по конфигурации
