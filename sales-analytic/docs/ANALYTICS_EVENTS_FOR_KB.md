# Использование analytics_events для обогащения Knowledge Base

## Обзор

Таблица `analytics_events` содержит реальные данные о поведении пользователей и может быть использована для автоматического обогащения и улучшения базы знаний (KB).

## Реализация

**Важно:** Функциональность реализована в реальных скриптах, а не в примерах кода в документации.

**Основной скрипт:** [`scripts/kb_enrichment.py`](../scripts/kb_enrichment.py)

**Документация по скриптам:** [`scripts/README.md`](../scripts/README.md)

**Конфигурация:** См. [`docs/CONFIGURATION_REFERENCE.md`](../../docs/CONFIGURATION_REFERENCE.md)

## Схема таблицы analytics_events

```sql
CREATE TABLE analytics_events (
    id SERIAL PRIMARY KEY,
    amocrm_lead_id INTEGER,  -- Связь с лидом в amoCRM
    event_type VARCHAR(50),  -- 'page_view', 'goal', 'hot_lead', 'conversation_message', ...
    event_data JSONB,  -- Детали события (URL, время, параметры, текст сообщения)
    created_at TIMESTAMP DEFAULT NOW()
);
```

**Типы событий, полезных для KB:**
- `conversation_message` — сообщения в диалогах с агентом
- `objection_detected` — обнаруженные возражения
- `question_asked` — вопросы клиентов
- `page_view` — просмотренные страницы (какие объекты интересуют)
- `goal` — достигнутые цели (какие действия совершают)

## Варианты использования analytics_events для KB

### 1. Обнаружение новых возражений

**Проблема:** В KB есть стандартные возражения, но клиенты могут задавать новые вопросы.

**Решение:** Анализировать события `objection_detected` и `conversation_message` для обнаружения новых паттернов возражений.

**Реализация:** Используется скрипт `scripts/kb_enrichment.py`

**Использование:**

```bash
# Анализ за последние 30 дней (dry-run - только анализ, без изменений)
python scripts/kb_enrichment.py --days 30 --dry-run

# Анализ за последние 7 дней с реальным добавлением в KB
python scripts/kb_enrichment.py --days 7
```

**Как работает:**

1. Скрипт анализирует события `objection_detected` за указанный период
2. Группирует возражения по типу и частоте (минимум 3 случая)
3. Проверяет, нет ли похожих возражений уже в KB
4. Для новых возражений генерирует ответы через LLM (используя модель из `config.env`)
5. Создает chunks в KB с автоматической генерацией embeddings (используя модель из `config.env`)

**Конфигурация:**

Скрипт использует переменные из `config.env`:
- `DATABASE_URL` - подключение к PostgreSQL
- `KB_API_URL` - URL KB Service API
- `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL` - для LLM (см. [CONFIGURATION_REFERENCE.md](../../docs/CONFIGURATION_REFERENCE.md))
- `EMBEDDING_PROVIDER`, `EMBEDDING_MODEL` или `HF_MODEL_NAME` - для embeddings (см. [CONFIGURATION_REFERENCE.md](../../docs/CONFIGURATION_REFERENCE.md))

**См. также:**
- [scripts/kb_enrichment.py](../scripts/kb_enrichment.py) - исходный код скрипта
- [scripts/README.md](../scripts/README.md) - документация по скриптам

### 2. Анализ популярных вопросов

**Проблема:** Клиенты задают вопросы, которых нет в KB.

**Решение:** Анализировать события `question_asked` и добавлять ответы в KB.

**Реализация:** Планируется в скрипте `scripts/analyze_questions.py` (в разработке)

**Статус:** ⏳ В разработке

**Планируемая функциональность:**

1. Анализ событий `question_asked` за указанный период
2. Группировка вопросов по категориям и частоте (минимум 5 раз)
3. Проверка наличия ответов в KB
4. Генерация ответов через LLM с использованием контекста из KB
5. Автоматическое добавление в KB с правильными метаданными

**См. также:**
- [scripts/README.md](../scripts/README.md) - документация по скриптам

### 3. Обновление информации о продуктах

**Проблема:** Клиенты интересуются объектами, которых нет в KB или информация устарела.

**Решение:** Анализировать события `page_view` и `goal` для понимания, какие объекты популярны.

**Реализация:** Планируется в скрипте `scripts/update_product_info.py` (в разработке)

**Статус:** ⏳ В разработке

**Планируемая функциональность:**

1. Анализ событий `page_view` для определения популярных объектов
2. Анализ вопросов по конкретным объектам
3. Проверка наличия информации в KB
4. Автоматическое добавление информации о популярных объектах
5. Обновление метаданных (добавление тега "популярный")

**См. также:**
- [scripts/README.md](../scripts/README.md) - документация по скриптам

### 4. Улучшение скриптов продаж

**Проблема:** Скрипты продаж могут быть неэффективными.

**Решение:** Анализировать конверсию по этапам диалога и обновлять скрипты.

**Реализация:** Планируется в скрипте `scripts/improve_sales_scripts.py` (в разработке)

**Статус:** ⏳ В разработке

**Планируемая функциональность:**

1. Анализ успешных диалогов (привели к КП)
2. Выявление паттернов успешных сообщений по этапам
3. Сравнение с текущими скриптами в KB
4. Автоматическое улучшение скриптов на основе успешных паттернов
5. Обновление скриптов в KB с версионированием

**См. также:**
- [scripts/README.md](../scripts/README.md) - документация по скриптам

### 5. Автоматическое обновление цен и наличия

**Проблема:** Информация о ценах и наличии может устаревать.

**Решение:** Анализировать события, связанные с ценами, и обновлять KB.

```python
async def detect_price_changes(db: asyncpg.Pool):
    """Обнаружение изменений цен на основе событий"""
    async with db.acquire() as conn:
        # Получение событий с ценами
        price_events = await conn.fetch("""
            SELECT 
                event_data->>'product_id' as product_id,
                event_data->>'price' as price,
                event_data->>'old_price' as old_price,
                created_at
            FROM analytics_events
            WHERE event_type = 'price_changed'
              AND created_at >= NOW() - INTERVAL '7 days'
            ORDER BY created_at DESC
        """)
        
        # Обновление информации в KB
        for event in price_events:
            await update_price_in_kb(conn, event["product_id"], event["price"])
```

## Преимущества использования analytics_events для KB

### ✅ Актуальность
- KB обновляется на основе реальных данных
- Новые возражения и вопросы добавляются автоматически

### ✅ Релевантность
- Фокус на том, что действительно интересует клиентов
- Популярные объекты получают больше внимания в KB

### ✅ Эффективность
- Скрипты продаж улучшаются на основе успешных диалогов
- Ответы на возражения становятся более эффективными

### ✅ Масштабируемость
- Автоматическое обогащение без ручной работы
- KB растёт вместе с количеством клиентов

## Примеры использования

### Пример 1: Обнаружение нового возражения

```
События в analytics_events:
- 5 раз клиенты спрашивали: "А что если дом не достроят?"

Анализ:
- Похожего возражения нет в KB
- Частота: 5 раз за месяц

Действие:
- Генерация ответа через LLM
- Добавление в KB с метаданными:
  - category: objection_handling
  - subcategory: construction_concern
  - source: analytics_events
  - frequency: 5
```

### Пример 2: Популярный вопрос

```
События в analytics_events:
- 20 раз клиенты спрашивали: "Можно ли купить в рассрочку?"

Анализ:
- В KB есть информация о финансировании, но не конкретно о рассрочке
- Частота: 20 раз за месяц

Действие:
- Генерация подробного ответа о рассрочке
- Добавление в KB с высоким приоритетом
```

### Пример 3: Улучшение скрипта

```
Анализ успешных диалогов:
- Диалоги, где агент упоминал "закрытая территория" в начале, имеют конверсию 40%
- Диалоги без этого упоминания: 25%

Действие:
- Обновление скрипта приветствия
- Добавление упоминания "закрытая территория" в начало диалога
```
