# Бизнес-процессы: чат, воронка продаж, DLP, AmoCRM

Единый workflow n8n реализует весь цикл: приём сообщения из чата (сайт/Telegram/Avito) → обезличивание для LLM → поиск в базе знаний → ответ нейропродажника → при наличии телефона создание лида и контакта в AmoCRM и записи в PostgreSQL для календаря.

---

## Стратегия: сначала n8n, потом FastAPI

**Сначала все сценарии реализуем и отлаживаем в n8n.** Когда бизнес-логика стабилизируется и станет ясной — продублируем её в FastAPI (sales-agent или отдельный API). Так экономим труд на отладке: один раз доводим до ума в workflow, затем переносим уже проверенное поведение в код.

- Текущий источник истины по потоку чата: **n8n** (workflow `sales-agent-kb-integration-localhost.json` и связанные).
- FastAPI (sales-agent, amocrm-api, funnel-api) — сервисы, которые n8n вызывает; дублирование полной логики диалога в FastAPI — следующий этап после стабилизации сценариев в n8n.

---

## 1. Вход и нормализация

- **Webhook:** `POST /webhook/sales-agent-kb`.
- **Тело запроса:** `message`, `channel`, `external_id`, `metadata` (в т.ч. `real_estate_context` — карточка объекта).
- **Normalize Input:** извлекает сообщение, канал, целевую аудиторию, формирует `additional_context` (название, цена, отделка, площадь и т.д.) для контекста LLM.

Исходное сообщение сохраняется в узле Normalize Input и используется **только** для извлечения телефона/имени при создании лида (Maybe Create Lead). В LLM уходит только обезличенный текст.

---

## 2. DLP (обезличивание перед LLM)

- **Цель:** не передавать ПДн в зарубежный LLM (proxyapi.ru / GPT-4o). Соответствие 152-ФЗ.
- **Узел DLP Sanitize Text:** `POST http://localhost:8001/api/dlp/sanitize-text`, body `{ "text": "<сообщение>" }`. Ответ: `{ "sanitized_text": "..." }`.
- **Merge Sanitized:** подставляет `sanitized_text` в поле `message` для дальнейшей цепочки. Поиск в KB и промпт для LLM строятся уже по обезличенному сообщению.
- Извлечение телефона/имени для AmoCRM выполняется из **исходного** сообщения (обращение к `$('Normalize Input')` в Maybe Create Lead).

## 3. Воронка продаж (стратегия в промпте)

Поведение агента задаётся системным промптом в узле Format KB Context.

- **Этап 1 — контакт не известен:** цель — получить имя и телефон, **а также email** (для КП и договора). В промпте явно предусмотреть запрос email в процессе диалога (например после телефона: «Укажите, пожалуйста, email — на него вышлем коммерческое предложение и документы»). Общая информация из базы, приглашение оставить контакты для расчёта и записи на просмотр.
- **Этап 2 — контакт известен:** ведение по воронке: запись на просмотр (слоты 9–18) → показ → КП/смета → согласование → ипотека → документы → оплата. Агент не останавливается после контакта: может предлагать просмотр, отделку (ссылка на сайт), КП, договор.
- **Диалоги из ЛК (`channel=lk`):** если объект недвижимости не определён в AmoCRM — уточнить, какой объект (участок, дом, планировка) интересует клиента. По ходу разговора уточнять параметры, соответствующие полям сделки в AmoCRM: бюджет, локация, вид отделки, первоначальный взнос, площадь, способ оплаты, ипотека, материнский капитал и т.д. Подробный список и формулировки для системного промпта: [docs/PERSONAL_CABINET_LK.md](../docs/PERSONAL_CABINET_LK.md) (раздел 5).

Ориентир воронки AmoCRM: Квалификация → Показ → КП/смета → Согласование проекта → Ипотека → Документы оформлены → Оплата → Успешно реализовано.

## 4. Календарь просмотров (слоты 9–18)

- **Узел Get Viewing Slots:** `GET http://localhost:8011/api/viewing-slots/available?days=7` (funnel-api). Возвращает свободные слоты.
- Результат объединяется с обезличенными данными в узле **Context With Slots** и передаётся в Format KB Context. В промпт добавляется блок «Свободные слоты для записи на просмотр (9–18)», если слоты есть.
- **Бронирование слота реализовано:** при фразе вида «24 февраля 10:00–11:00 запишите» срабатывают узлы **Parse Slot From Message** (парсинг даты/времени из сообщения) → **Book Slot If Parsed** (условие по `should_book`) → **Book Viewing Slot API** (`POST http://localhost:8011/api/viewing-slots/book`). Запись создаётся в таблице `viewing_slots` (funnel-api, PostgreSQL). Подробнее: [docs/FUNNEL_CALENDAR_CP_CONTRACT.md](../docs/FUNNEL_CALENDAR_CP_CONTRACT.md).

Перед использованием календаря нужно создать слоты: `POST http://localhost:8011/api/viewing-slots/seed?days=7`.

## 5. База знаний и LLM

- **KB Search:** поиск по обезличенному сообщению и `additional_context` в kb-service (`POST /api/kb/search`). Результаты попадают в блок «База знаний» промпта.
- **Format KB Context:** формирует системный промпт (воронка, правила) и пользовательский контент: база знаний + контекст канала + при наличии — слоты + обезличенное сообщение клиента. **Обязательно:** если в ответе Resolve есть `slots` с полями `client_name` (или `name`), `phone`, `email` — добавить в системный промпт блок вида: «Известные данные клиента: имя — {{slots.client_name}}, телефон — {{slots.phone}}, email — {{slots.email}}. Не переспрашивай имя и контакты, если они уже указаны; обращайся по имени.» Тогда агент не будет спрашивать «Уточните имя» после заявки с формы. В системном промпте также заложить: после получения телефона (или вместе с ним) запрашивать **email**, если его ещё нет в слотах — «Укажите email, чтобы мы могли выслать коммерческое предложение и документы».
- **LLM ProxyAPI:** вызов GPT-4o через proxyapi.ru. Ответ возвращается клиенту через Respond to Webhook.

## 6. Создание лида и контакта в AmoCRM

- **Maybe Create Lead:** из **исходного** сообщения (Normalize Input) извлекаются **телефон**, при возможности **имя** и **email**. Обязательно распознавать форматы: «тел. 79106337788», «79106337788», «+7 910 633 77 88», «9106337788» — иначе лид в CRM не создастся (агент ответит по имени из текста, но контакт не появится). Готовая логика и пример кода для узла: [MAYBE_CREATE_LEAD_EXTRACTION.md](MAYBE_CREATE_LEAD_EXTRACTION.md). Если телефон найден — `should_create: true`, формируется `lead_payload`: **phone** (нормализованные цифры), **name**, **email** (если извлекли), **note**, **external_id**, **channel**.
- **Create Lead If Phone:** условие по `should_create`.
- **AmoCRM Create Lead:** `POST http://localhost:8010/api/test-lead-from-chat` с телом `lead_payload`. amocrm-api создаёт контакт и лид в AmoCRM, записывает телефон и email в поля контакта/сделки (ID полей заданы в config.env: `AMOCRM_CONTACT_PHONE_FIELD_ID`, `AMOCRM_CONTACT_EMAIL_FIELD_ID` и при необходимости lead-поля). При наличии DATABASE_URL пишет запись в таблицу **conversations** и возвращает `conversation_id` для календаря и КП.

## 6.1. Связь клиента с менеджером: запрос осмотра и передача (handoff)

В цепочке n8n предусмотреть **два варианта**, когда клиента нужно связать с менеджером.

### Случай 1: Запрос осмотра (запись на просмотр)

- **Что уже есть:** при фразе вида «24 февраля 10:00–11:00 запишите» вызывается **Book Viewing Slot API** (funnel-api), создаётся запись в `viewing_slots` со статусом `booked`, привязкой к `conversation_id` и при наличии — `amocrm_lead_id`.
- **Дополнительно (новейшие варианты в документации):** после успешного бронирования слота **создать задачу или примечание в AmoCRM** по лиду, чтобы менеджер видел предстоящий показ: дата, время, контакт (телефон/имя из слотов или из `conversations.slots`). Так менеджер получает явное уведомление и может подготовиться к осмотру. В n8n: после узла **Book Viewing Slot API** добавить ветку (или подзапрос) → HTTP Request к AmoCRM API: создание задачи к сделке (`POST /api/v4/tasks` с `entity_id = amocrm_lead_id`, текст задачи «Показ: дата, время, объект») или примечание к лиду (`POST /api/v4/leads/{id}/notes` с текстом «Запись на просмотр: …»). Данные для задачи/примечания брать из ответа Book Slot и из Merge Resolve (conversation_id → при необходимости запрос к amocrm-api за lead_id по conversation_id).
- См. [docs/FUNNEL_CALENDAR_CP_CONTRACT.md](../docs/FUNNEL_CALENDAR_CP_CONTRACT.md) (календарь, слоты) и [docs/AMOCRM.md](../docs/AMOCRM.md) (задачи менеджеру).

### Случай 2: Прямой запрос «связать с менеджером» и другие просьбы к человеку

- **Когда срабатывает:** клиент пишет «хочу поговорить с менеджером», «свяжите с человеком», «позовите оператора», «нужен живой консультант» и т.п. (Human in the loop, HITL).
- **Правила (по документации):** не отговаривать; подтвердить передачу; если контакт уже есть (телефон, ФИО) — в ответе клиенту явно сказать: *«Передаю вашу заявку менеджеру. Он свяжется с вами по указанному номеру / в ближайшее время.»*
- **Что сделать в системе:**
  1. Обновить `conversations.state` на `HANDOFF` (через API amocrm-api, когда будет эндпоинт `POST /api/conversation/{id}/handoff`, или прямым обновлением в БД из n8n).
  2. Создать **задачу ответственному в AmoCRM** по лиду/контакту (если есть `amocrm_lead_id` — из Resolve или из conversations).
  3. Добавить **примечание к сделке** с резюме диалога (тема, что обсудили, контактные данные), чтобы менеджер видел контекст.
- **В цепочке n8n:** после ответа LLM (или параллельно) — узел распознавания интента по ключевым фразам («менеджер», «человек», «оператор», «поговорить с вами», «живой консультант»). При срабатывании — вызов API передачи: обновление state + создание задачи в AmoCRM + примечание. Ответ клиенту формировать с подтверждением передачи (можно подставлять фиксированный ответ при handoff вместо ответа LLM или дополнять им).
- Подробно: [docs/FUNNEL_CALENDAR_CP_CONTRACT.md](../docs/FUNNEL_CALENDAR_CP_CONTRACT.md), раздел 5.1 «Human in the loop»; [docs/BUSINESS_PROCESSES.md](../docs/BUSINESS_PROCESSES.md) — FSM и переход в HANDOFF, маршрутизация к менеджеру.

**Итог:** при **запросе осмотра** — кроме записи в календарь, дать менеджеру задачу/примечание по сделке; при **явном запросе связать с менеджером** — handoff: state HANDOFF, задача в AmoCRM, примечание с резюме, ответ клиенту с подтверждением.

## 7. Ответ клиенту

- **Prepare Response** собирает ответ (поле `response` / `agent_response`) и при необходимости метаданные.
- **Respond to Webhook** возвращает JSON с ответом в чат (сайт/Telegram/Avito).

## 8. Память диалога и сохранение реплик (обязательно для адекватного агента)

Без памяти сессии агент переспрашивает имя, телефон, выбор слота и ведёт себя бессвязно. **Поддержка памяти в текущей сессии обязательна.**

### Как обеспечивается память

1. **Серверная память (рекомендуется):** чат при каждом запросе отправляет **external_id** (идентификатор сессии). По нему workflow загружает историю из PostgreSQL и после ответа сохраняет новую пару реплик.
   - **Откуда брать external_id:** сгенерировать на фронте при открытии чата (например `uuid` или `chat-${Date.now()}-${random}`), сохранить в `localStorage` / cookie / state и передавать в теле каждого POST на webhook: `{ "message": "...", "external_id": "<тот же id>" }`.
   - Требования: миграция `09_add_conversation_external_id.sql` применена; amocrm-api доступен на 8010.

2. **Резервный вариант (если нет external_id или API недоступен):** чат может передавать в теле **conversation_history** — массив последних реплик `[{ "role": "user", "content": "..." }, { "role": "assistant", "content": "..." }]`. Merge Resolve подставит его, если от API пришла пустая история. Так агент хотя бы видит контекст в рамках одного «окна» (клиент хранит историю у себя и шлёт с каждым сообщением).

### Узлы

- **Resolve Conversation:** (Code node) GET к amocrm-api за историей диалога. Базовый URL берётся из переменной окружения **AMOCRM_API_BASE_URL** (без слеша в конце); если не задана — `http://localhost:8010`. Менять хост/порт нужно только в этой переменной в n8n, не в коде workflow. API возвращает `conversation_id`, `messages`, **`slots`** (имя, телефон, email и др.) и `state`. Слоты нужны, чтобы агент не переспрашивал имя и контакты (см. ниже).
- **Merge Resolve:** склеивает нормализацию с `conversation_id`, `conversation_history` и **`slots`** из ответа Resolve. При ошибке API подставляет `conversation_id: null` и при наличии — `conversation_history` из тела запроса (fallback).
- **Save Messages Prep** + **Save Messages API:** при наличии `conversation_id` сохраняют реплику user и ответ assistant в PostgreSQL. При ошибке — onError continue.

**Итог:** для стабильной памяти чат должен передавать **external_id**. Дополнительно можно передавать **conversation_history** как подстраховку.

## 9. Отладка текущего workflow (без новых веток)

Чеклист для завершения отладки **текущего** потока (существенные новые ветки не добавляем):

1. **Сервисы:** kb-service (8001), funnel-api (8011), amocrm-api (8010) — подняты; n8n (5678) запущен.
2. **DLP:** `curl -s http://localhost:8001/api/dlp/health` → ok; POST sanitize-text с текстом с телефоном → в ответе маска.
3. **Слоты:** `curl -s "http://localhost:8011/api/viewing-slots/available?days=7"` → массив слотов (или пусто); при пустом ответе workflow не падает (Get Viewing Slots → onError continue, Context With Slots даёт `slots: []`).
4. **Память:** миграция `09_add_conversation_external_id.sql` применена; при запросе с `external_id` в теле — Resolve возвращает conversation_id и messages; при падении amocrm-api — поток не рвётся (Merge Resolve отдаёт null и []).
5. **Полный проход:** POST на webhook с телом `{ "message": "Привет", "external_id": "test-1" }` → ответ от бота; повторный запрос с тем же `external_id` и другим сообщением → ответ с учётом контекста (если память и LLM работают).
6. **Лид:** запрос с телефоном в сообщении → создаётся лид в AmoCRM (если amocrm-api настроен); ответ клиенту приходит в любом случае.
7. **Сохранение реплик:** при наличии `conversation_id` после ответа вызывается Save Messages API; при его ошибке ответ клиенту не страдает.
8. **Запись в календарь:** при сообщении вида «24 февраля 10:00–11:00 запишите» (и наличии `conversation_id`) вызывается Book Viewing Slot API; в БД создаётся запись в `viewing_slots`. Проверка: `SELECT * FROM viewing_slots WHERE status = 'booked' ORDER BY created_at DESC LIMIT 5;`

План полного прохода клиента по воронке и завершения тестирования: см. [ПЛАН_ТЕСТИРОВАНИЯ_ВОРОНКИ.md](ПЛАН_ТЕСТИРОВАНИЯ_ВОРОНКИ.md).

## Сводка потоков данных

| Данные | Куда идут | DLP |
|--------|-----------|-----|
| Сообщение пользователя | В LLM (в промпте) | Да (sanitized_text) |
| Сообщение пользователя | Извлечение телефона/имени для AmoCRM | Нет (исходное из Normalize Input) |
| additional_context | В промпт LLM | Не требуется (карточка объекта) |
| Слоты календаря | В промпт LLM (если есть) | Не требуются |
| Результаты KB Search | В промпт LLM | Уже обезличены в KB при необходимости |
