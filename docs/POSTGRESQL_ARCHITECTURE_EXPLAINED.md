# Архитектура PostgreSQL таблиц для Нейропродажника

## Краткий ответ

**Таблица `products` — это НЕ просто кэш, а основной инструмент агента с векторным поиском.**

Векторный поиск используется **прямо в таблице `products`** через поле `description_embedding`. Это не дополнительный кэш, а полноценная система поиска объектов для агента.

---

## Детальное объяснение

### Две системы векторного поиска

В PostgreSQL есть **две отдельные таблицы** с векторным поиском, каждая для своей задачи:

#### 1. `knowledge_base` — База знаний (KB)
**Назначение:** Скрипты продаж, обработка возражений, tone of voice, контакты

```sql
CREATE TABLE knowledge_base (
    id SERIAL PRIMARY KEY,
    content TEXT,  -- Текстовая информация (скрипты, ответы на возражения)
    embedding vector(1536),  -- Векторное представление content
    metadata JSONB  -- Категория, целевая аудитория, приоритет
);
```

**Использование:**
- Агент ищет **как отвечать** на вопросы клиента
- Пример: "Что сказать, если клиент говорит 'это дорого'?"
- Поиск: `SELECT * FROM knowledge_base WHERE embedding <=> $query_vector`

#### 2. `products` — Каталог объектов недвижимости
**Назначение:** Поиск конкретных объектов по запросу клиента

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200),  -- "Коттедж BLACK BOX, участок 12"
    description TEXT,  -- Полное описание объекта
    description_embedding vector(1536),  -- Векторное представление description
    price_current DECIMAL(12, 2),
    -- ... другие поля
);
```

**Использование:**
- Агент ищет **какие объекты** предложить клиенту
- Пример: "Мне нужен дом с 3 спальнями, бюджет до 10 млн"
- Поиск: `SELECT * FROM products WHERE description_embedding <=> $query_vector`

## Архитектура: amoCRM → PostgreSQL → Агент

### Шаг 1: Синхронизация из amoCRM

```
amoCRM каталог (nomenklatura)
    ↓
n8n workflow: синхронизация
    ↓
PostgreSQL products
    ├─→ Базовые данные (name, price, SKU) из amoCRM
    ├─→ Генерация embedding для description
    └─→ Сохранение в description_embedding
```

**Что синхронизируется:**
- Базовые данные: название, цена, SKU, описание
- Генерируется embedding для векторного поиска
- Добавляются дополнительные поля (advantages, objections_handling)

### Шаг 2: Использование агентом

```
Клиент: "Мне нужен дом с 3 спальнями"
    ↓
Агент генерирует embedding для запроса
    ↓
Векторный поиск в products.description_embedding
    ↓
Находит подходящие объекты
    ↓
Использует для генерации КП
```

## Почему не просто кэш?

### ❌ Неправильное понимание:
> "products — это кэш из amoCRM для удобства агента"

### ✅ Правильное понимание:
> "products — это расширенная версия каталога с векторным поиском, специально созданная для агента"

### Ключевые отличия от простого кэша:

1. **Векторный поиск** — основная функция
   - Поле `description_embedding` позволяет семантический поиск
   - Агент может найти объекты по смыслу, а не только по точному совпадению

2. **Дополнительные данные для RAG**
   - `advantages` — массив преимуществ для обработки возражений
   - `objections_handling` — готовые ответы на возражения по объекту
   - `features` — структурированные характеристики в JSONB

3. **Связь с другими таблицами**
   - `discount_rules` — варианты скидок
   - `sales_options` — варианты продаж (отделка, комплектация)
   - `product_sales_options` — связь объектов с вариантами

4. **Оптимизация для агента**
   - Индексы для быстрого векторного поиска (IVFFlat)
   - Индексы для фильтрации по цене, категории, статусу
   - Структура данных оптимизирована для запросов агента

## Как работает векторный поиск в products

### Пример запроса агента:

```python
# Клиент: "Мне нужен дом с 3 спальнями, бюджет до 10 млн"

# 1. Генерация embedding для запроса
query_embedding = openai.embeddings.create(
    model="text-embedding-3-small",
    input="Мне нужен дом с 3 спальнями, бюджет до 10 млн"
).data[0].embedding

# 2. Векторный поиск в products
results = await db.fetch("""
    SELECT 
        id, name, description, price_current, rooms_count,
        1 - (description_embedding <=> $1::vector) as similarity
    FROM products
    WHERE status = 'available'
      AND price_current <= 10000000
      AND 1 - (description_embedding <=> $1::vector) > 0.6
    ORDER BY description_embedding <=> $1::vector
    LIMIT 5
""", query_embedding)
```

**Что происходит:**
1. Запрос клиента преобразуется в вектор (embedding)
2. PostgreSQL ищет объекты с похожими векторами в `description_embedding`
3. Находит объекты, где описание семантически похоже на запрос
4. Фильтрует по цене и другим критериям
5. Возвращает топ-5 наиболее релевантных объектов

## Сравнение: amoCRM каталог vs PostgreSQL products

| Характеристика | amoCRM каталог | PostgreSQL products |
|----------------|----------------|---------------------|
| **Назначение** | Управление номенклатурой для менеджеров | Поиск объектов для агента |
| **Векторный поиск** | ❌ Нет | ✅ Есть (description_embedding) |
| **Семантический поиск** | ❌ Нет | ✅ Есть (по смыслу, не только по ключевым словам) |
| **Дополнительные данные** | ❌ Только базовые поля | ✅ advantages, objections_handling, features |
| **Связь со скидками** | ❌ Нет | ✅ Есть (discount_rules) |
| **Связь с вариантами** | ❌ Нет | ✅ Есть (sales_options) |
| **Оптимизация для агента** | ❌ Нет | ✅ Да (индексы, структура) |
| **Синхронизация с 1C** | ✅ Есть (CSV) | ❌ Нет (синхронизируется из amoCRM) |

## Процесс работы агента с products

### 1. Поиск объектов для КП

```
Клиент: "Мне нужен дом с 3 спальнями"
    ↓
Агент: векторный поиск в products.description_embedding
    ↓
Находит объекты с похожим описанием
    ↓
Фильтрует по критериям (цена, площадь, комнаты)
    ↓
Предлагает клиенту варианты
```

### 2. Обработка возражений

```
Клиент: "Этот дом дорого стоит"
    ↓
Агент читает objections_handling для объекта из products
    ↓
Или использует advantages для формирования ответа
    ↓
Предлагает альтернативы или варианты оплаты
```

### 3. Генерация КП

```
Агент определил подходящие объекты
    ↓
Использует данные из products (цена, описание, характеристики)
    ↓
Создаёт КП с товарами
    ↓
Добавляет товары в сделку amoCRM через API
    ↓
amoCRM автоматически рассчитывает стоимость
```

## Ответы на вопросы

### ❓ Это кэш объектов из amoCRM?

**Частично да, но не только:**
- Да, базовые данные синхронизируются из amoCRM
- Нет, это не просто кэш — это расширенная версия с векторным поиском
- Это основной инструмент агента для поиска объектов

### ❓ Это тулзы для агента или векторка отдельно?

**Векторный поиск используется ПРЯМО в таблице products:**
- Поле `description_embedding` содержит векторное представление описания
- Агент выполняет векторный поиск прямо в этой таблице
- Это не отдельная система, а встроенная функция таблицы

### ❓ Кэш дополнительно?

**Нет, это не дополнительный кэш:**
- Это основной инструмент агента для работы с объектами
- Векторный поиск — основная функция, а не дополнение
- Без этой таблицы агент не сможет искать объекты по смыслу
