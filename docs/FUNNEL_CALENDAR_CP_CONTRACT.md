# Воронка после контакта: просмотр, КП, договор

После того как клиент оставил контакт (имя, телефон), агент **продолжает вести диалог** и ведёт по воронке продаж.

**Проиграть сценарий «повторный заход → КП → согласование → окончательный договор»:** пошагово описано в [SCENARIO_REPEAT_VISIT_CP_CONTRACT.md](./SCENARIO_REPEAT_VISIT_CP_CONTRACT.md) (API, чек-лист, интеграция с n8n). Не останавливается, даже если просмотр не заказан: обсуждает отделку, КП, формы договора, финансы.

## 1. Сценарии после контакта

| Запрос клиента | Действие агента | Узлы / API / PostgreSQL |
|----------------|-----------------|--------------------------|
| **Запись на просмотр** | Предложить свободные слоты (рабочее время 9–18), записать выбранный слот | Календарь: `viewing_slots` (список свободных, бронирование) |
| **Варианты отделки** | Дать ссылку на сайт (каталог отделки), кратко описать по базе знаний | Без отдельного узла — ответ из KB + ссылка в промпте |
| **КП (коммерческое предложение)** | Подготовить/отправить КП по объекту и контакту | КП: `proposals`, `document_templates` (type=proposal) |
| **Договор, формы** | Объяснить условия, дать ссылку на шаблон или форму | Договор: `document_templates` (type=contract), при необходимости генерация по шаблону |
| **Общие вопросы (ипотека, оплата, сроки)** | Отвечать по базе знаний, вести к следующему шагу воронки | Текущий поток KB + LLM |
| **«Хочу поговорить с человеком» / «свяжите с менеджером» (Human in the loop)** | Подтвердить передачу менеджеру, не отговаривать. Если у клиента уже есть телефон и ФИО — передать их в задачу/лид и чётко сообщить: «Менеджер свяжется с вами по указанному контакту». | `conversations.state = HANDOFF`, создание задачи в AmoCRM, примечание/резюме диалога для менеджера |

Агент не «заканчивает» после контакта: может одновременно предложить и просмотр, и ссылку на отделку, и КП — в зависимости от контекста.

## 2. PostgreSQL: какие таблицы задействованы

| Таблица | Назначение |
|---------|------------|
| **conversations** | Диалог, контактные данные (slots: phone, client_name, email), связь с AmoCRM (`amocrm_lead_id`, `amocrm_contact_id`). Заполняется при создании лида из чата (amocrm-api). |
| **viewing_slots** | Календарь показов: слоты 9–18, статусы free/booked/completed/cancelled, привязка к `conversation_id`, `amocrm_lead_id`, объект (object_id/products). |
| **proposals** | КП: версии, статусы (draft/sent/finalized/accepted), связь с `conversation_id`, `amocrm_lead_id`, структура (content_structured), product_ids. |
| **document_templates** | Шаблоны КП и договоров (type=proposal, type=contract), body/body_structured. |
| **products** | Объекты (дома, участки) — для слотов и КП. |

Схемы: `db-init/03_create_conversations.sql`, `db-init/06_create_proposals.sql`, `db-init/07_viewing_slots_and_templates.sql`.

## 3. Узлы и API для n8n и агента

Чтобы агент мог «выйти на просмотр», «отправить КП», «дать договор», нужны вызовы из n8n (или из чата) к сервису, который работает с PostgreSQL.

### 3.1 Календарь (встреча / просмотр)

- **Рабочее время:** 9:00–18:00, рабочие дни (пока без учёта праздников — можно задать в конфиге).
- **Узлы (n8n):**
  - Перед ответом LLM: при наличии в сообщении интента «запись на просмотр» / «хочу посмотреть» — вызвать API свободных слотов.
  - После выбора клиентом времени — вызвать API бронирования слота.
- **API (funnel-api или общий сервис с БД):**
  - `GET /api/viewing-slots/available` — свободные слоты на ближайшие N дней (9–18), опционально по object_id, settlement_id. Возврат: список { slot_start, slot_end, id }.
  - `POST /api/viewing-slots/book` — тело: `conversation_id`, `slot_start`, `slot_end`, `contact_name`, `contact_phone`, `amocrm_lead_id?`, `object_id?`, `notes?`. Создаёт запись в `viewing_slots` со статусом `booked`, привязывает к `conversation_id`.
- **Заполнение календаря:** слоты на период (например неделя вперёд 9–18) создаются заранее (скрипт/задача или при первом запросе available). Статус `free` по умолчанию.

### 3.2 КП (коммерческое предложение)

- **Узлы:** при запросе «хочу КП», «пришлите предложение» — вызов API создания/получения КП по conversation_id/lead_id, возврат ссылки или текста для ответа клиенту.
- **API (funnel-api, порт 8011):**
  - `GET /api/proposals?conversation_id=` — список КП по диалогу.
  - `POST /api/proposals` — создать черновик КП (тело: conversation_id, amocrm_lead_id?, product_ids?, template_id?). При указании template_id подставляется body шаблона в content_text. Возврат: id, status, created_at.
  - `GET /api/proposals/{id}` — один КП (content_text для выдачи клиенту).
  - `GET /api/document-templates?type=proposal` — список активных шаблонов КП.

### 3.3 Договор

- **Узлы:** при вопросах про договор, форму, условия — вызов API шаблонов договора или выдача ссылки на страницу с формой/документом.
- **API:**
  - `GET /api/document-templates?type=contract` — список шаблонов договора.
  - При необходимости: `POST /api/contracts/generate` — генерация документа по шаблону и данным из conversation/lead (позже).

## 4. Интеграция с n8n и промптом агента

- **Системный промпт (Format KB Context):** добавить явно:
  - После контакта продолжать диалог: предлагать запись на просмотр (слоты 9–18), при интересе к отделке — ссылка на сайт, при запросе КП/договора — следующий шаг (КП/договор).
  - Не останавливаться на одном шаге: можно одновременно обсудить и просмотр, и КП, и условия договора.
- **Ветки в workflow:**
  - Из «Prepare Response» или из отдельного узла распознавания интента: если «просмотр» → HTTP Request к `GET /api/viewing-slots/available`, результат подставить в контекст LLM или в ответ («Вот свободные окна: …»).
  - Если пользователь выбрал время (парсинг из сообщения) → `POST /api/viewing-slots/book`, затем ответ «Вы записаны на …».
  - Аналогично для КП: при интенте «КП» → вызов API КП, в ответ — ссылка или текст.
  - Для договора: при интенте «договор» → шаблоны/ссылка.

- **Контекст для LLM:** в блок «Контекст от канала» можно добавлять структурированные данные:
  - `available_slots: [{ date, time_start, time_end }]` — если запрашивали слоты;
  - `proposal_link: "…"` или `contract_info: "…"` — если уже сформировали КП/договор.

## 5. Дальнейшие шаги воронки: выбор объекта, резерв, оплата

После просмотра клиент может **выбрать конкретный дом (адрес)** для покупки. Логика та же, что с календарём: действие агента должно быть осмысленным и подтверждаться в системе.

| Этап | Что делает клиент | Что делает агент / система | Где фиксируется |
|------|-------------------|----------------------------|------------------|
| **Выбор объекта** | Выбирает дом/участок по адресу или из каталога | Фиксирует выбор, привязывает к диалогу и лиду | `proposals` (product_ids), при необходимости — отдельная сущность «резерв объекта» |
| **Резерв** | Соглашается зарезервировать объект (в рамках КП или договора) | Резервирует объект на срок (как слот в календаре, но для объекта), формирует КП или проект договора | КП: `proposals` + статус/резерв; договор: черновик в `document_templates` или отдельная таблица договоров с привязкой к объекту |
| **Оплата (исполнение договора)** | Оплачивает по договору | Фиксирует оплату, переводит сделку в статус «исполнено», снимает резерв с объекта | Статус договора/сделки в БД, обновление AmoCRM, при необходимости — интеграция с платёжкой |

Идея: **резерв объекта** — по смыслу аналог записи в календарь, но не на «просмотр», а на «покупку»: клиент выбрал адрес дома, агент зарезервировал его (в КП или в договоре), затем клиент оплачивает и договор исполняется. Агент должен явно подтверждать клиенту: что зарезервировано, на какой срок, что делать дальше (оплата, подписание).

- **В КП:** резерв может быть отражён в статусе КП (например «объект зарезервирован до даты X») и в `proposals.content_structured` или в отдельной таблице резервов по `product_id` + `conversation_id` / `amocrm_lead_id`.
- **В договоре:** резерв/выбор объекта фиксируется в проекте договора (конкретный объект, адрес), после подписания и оплаты — статус «договор исполнен».

Реализация (позже): API «резерв объекта», расширение `proposals` или отдельная таблица `object_reservations`, узлы в n8n по аналогии с календарём (создать резерв → подтвердить клиенту).

## 5.1. Human in the loop (передача клиента человеку)

Когда клиент просит **связать с человеком** («хочу поговорить с менеджером», «позовите оператора», «свяжите с живым консультантом») — это сценарий **human in the loop (HITL)**. Его нужно предусмотреть явно.

**Правила:**

1. **Не отговаривать** — не предлагать «сначала ответьте на пару вопросов». Если клиент просит человека, подтвердить и инициировать передачу.
2. **Особенно важно, если контакт уже есть:** если клиент уже дал **телефон и ФИО**, передача менеджеру максимально осмысленна: лид/контакт уже в AmoCRM, менеджер может сразу позвонить или написать. В ответе клиенту обязательно сказать: *«Передаю вашу заявку менеджеру. Он свяжется с вами по указанному номеру / в ближайшее время.»*
3. **Что сделать в системе:** обновить `conversations.state` на `HANDOFF`, создать задачу ответственному в AmoCRM по лиду/контакту, при необходимости добавить примечание с резюме диалога (тема, что обсудили, контактные данные).

**Технически:** в n8n после ответа LLM можно по интенту/ключевым фразам («менеджер», «человек», «оператор», «поговорить с вами») вызывать API передачи (например `POST /api/conversation/{id}/handoff` в amocrm-api или обновление state в БД + создание задачи в AmoCRM). В ответе клиенту — только подтверждение передачи и ожидание звонка/письма от менеджера.

**Связь с менеджером в двух случаях (запрос осмотра и явный handoff):** в [n8n/BUSINESS_PROCESSES.md](../n8n/BUSINESS_PROCESSES.md), раздел **6.1**, описаны оба варианта: (1) при **запросе осмотра** — после бронирования слота создавать задачу/примечание в AmoCRM по сделке, чтобы менеджер видел предстоящий показ; (2) при **прямом запросе** «связать с менеджером» — handoff (state HANDOFF, задача, примечание с резюме).

## 5.2. Высылка договора по email и закрытие сделки

Полный сюжет воронки до конца: после согласования КП клиент запрашивает **договор на email**; система отправляет письмо с договором и переводит сделку в статус «Успешно реализовано» (или оставляет менеджеру закрытие после подписания).

| Шаг | Действие | API / n8n |
|-----|----------|-----------|
| 1 | Клиент: «Вышлите договор на почту» / «Готов подписать, пришлите на email» | Интент «договор на email» в n8n |
| 2 | Проверка email | Email берётся из `conversations.slots.email` (при создании лида из чата или из слотов диалога). Если нет — ответ: «Укажите, пожалуйста, ваш email для отправки договора» и запись в слоты. |
| 3 | Подготовка письма | **funnel-api:** `POST /api/contracts/prepare-email` с телом `{ "conversation_id": N, "template_id": опционально, "to_email_override": опционально }`. Возвращает `to_email`, `subject`, `body` (шаблон договора с подстановкой {{client_name}}, {{email}}, {{phone}} из слотов). |
| 4 | Отправка письма | В n8n: узел **Send Email** (SMTP или почтовый API), получатель и тело — из ответа prepare-email. Либо HTTP Request к внешнему сервису отправки писем. |
| 5 | Закрытие сделки | **amocrm-api:** `POST /api/leads/{lead_id}/close` (опционально `?status_id=142`). Переводит сделку в этап «Успешно реализовано» (142 в воронке «Работы маркетинг», см. [AMOCRM.md](./AMOCRM.md)). Вызывать из n8n после успешной отправки письма или по отдельному интенту «договор подписан, закрыть сделку». |
| 6 | Подтверждение клиенту | Ответ в чате: «Договор отправлен на указанный email. После подписания мы свяжемся с вами для дальнейших шагов.» При закрытии сделки можно добавить примечание к лиду в AmoCRM (договор отправлен, дата). |

**Где хранить факт «договор отправлен»:** примечание к сделке в AmoCRM (`add_lead_note`) или поле в БД (например в `proposals` — contract_sent_at при расширении схемы). Для минимальной реализации достаточно примечания к лиду и перевода в закрытый статус.

**Конфигурация:** в config.env для amocrm-api можно задать `AMOCRM_CLOSED_SUCCESS_STATUS_ID=142` (ID этапа «Успешно реализовано» в вашей воронке).

## 6. Порядок внедрения

1. **Календарь (просмотр):** API свободных слотов + бронирование (funnel-api или отдельный сервис), сидирование слотов 9–18 на неделю вперёд. n8n: узел запроса слотов и узел бронирования по ответу пользователя.
2. **Промпт и ветки:** обновить системный промпт; добавить ветку «просмотр» с вызовом API слотов и бронирования.
3. **КП:** API списка/создания КП по conversation_id, шаблоны из document_templates. В n8n — узел «КП» с вызовом API и подстановкой ссылки в ответ.
4. **Договор:** API шаблонов договора + при необходимости страница/ссылка на форму. Узел «договор» в n8n.
5. **Высылка договора по email и закрытие сделки:** см. раздел 5.2 — `POST /api/contracts/prepare-email` (funnel-api), узел Send Email в n8n, `POST /api/leads/{id}/close` (amocrm-api).
6. **Дальше:** резерв выбранного объекта (в КП или в договоре), затем оплата и исполнение договора — см. раздел 5 выше.
7. **Human in the loop:** при запросе «связать с менеджером» — обновление `conversations.state = HANDOFF`, создание задачи в AmoCRM, ответ клиенту с подтверждением; при наличии телефона и ФИО — явно указать в задаче/примечании и в ответе клиенту, что менеджер свяжется по этому контакту (см. раздел 5.1).

Сервис **funnel-api** (порт 8011) реализует эти API, использует DATABASE_URL из config.env. Запуск: `./start_all_services.sh` (поднимает funnel-api вместе с остальными) или вручную: `cd funnel-api && uvicorn api.main:app --host 0.0.0.0 --port 8011`.

- **Перед первым использованием календаря:** вызвать `POST http://localhost:8011/api/viewing-slots/seed?days=7` — создаст свободные слоты 9–18 на неделю вперёд.
