# Интеграция с amoCRM: API, примеры и бизнес-процессы

Единый документ: настройка доступа, описание API **amocrm-api**, примеры использования, выводы о бизнес-процессах. **OAuth2 и получение токенов:** [AMOCRM_API_SETUP.md](./AMOCRM_API_SETUP.md). Оглавление документации: [docs/README.md](./README.md).

## 1. Настройка доступа

**Переменные** задаются в **config.env в корне репозитория** (`/projects/brats/config.env`). Их читают amocrm-api (порт 8010), скрипты `scripts/check_amocrm_*.py` (они подгружают `../config.env`) и при необходимости n8n (если запущен через `./start_all_services.sh`, он делает `source config.env`). В шаблоне **config.env.example** есть закомментированная секция AmoCRM — раскомментируйте и подставьте значения.

| Переменная | Назначение |
|------------|------------|
| `AMOCRM_SUBDOMAIN` | Поддомен (например, innovatoryclub) |
| `AMOCRM_ACCESS_TOKEN` | Токен доступа (обновляется по refresh) |
| `AMOCRM_REFRESH_TOKEN` | Токен обновления |
| `AMOCRM_CLIENT_ID`, `AMOCRM_CLIENT_SECRET`, `AMOCRM_REDIRECT_URI` | Для автообновления токена |
| `AMOCRM_STATUS_KP_SENT` | (Опционально.) ID этапа «КП/смета отправлена». Задайте в config.env в **корне репо** — funnel-api при создании КП переведёт сделку на этот этап |
| `AMOCRM_STATUS_CONTRACT_SENT` | (Опционально.) ID этапа «Договор/документы отправлены». Задайте в config.env в **корне репо** — funnel-api при подготовке письма с договором переведёт сделку на этот этап |

Токены можно получить через n8n или по шагам из [AMOCRM_API_SETUP.md](./AMOCRM_API_SETUP.md). Если вы уже тестировали скрипты check_amocrm_* и переменные были в другом файле — перенесите их в **config.env** в корне репозитория и перезапустите amocrm-api.

**PostgreSQL (для записи в календарь):** если в config.env задан **DATABASE_URL** (тот же, что для KB), то при создании лида из чата amocrm-api дополнительно пишет строку в таблицу **conversations** (amocrm_lead_id, amocrm_contact_id, channel, slots с именем/телефоном/email) и возвращает в ответе **conversation_id**. Он нужен для последующей **записи в календарь** (таблица `viewing_slots` ссылается на `conversation_id`). Без DATABASE_URL лид и контакт создаются только в AmoCRM, `conversation_id` в ответе будет `null`.

## 2. Описание API (amocrm-api)

Сервис в директории **amocrm-api/**: FastAPI, чтение amoCRM и создание тестовых лидов. Токен подхватывается из корня репо, при 401 обновляется по refresh.

### Запуск

```bash
cd amocrm-api
pip install -r requirements.txt
Жuvicorn api.main:app --reload --host 0.0.0.0 --port 8010
```

Swagger: `http://localhost:8010/docs`

### Эндпоинты

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/health` | Проверка работы сервиса |
| GET | `/api/pipelines` | Воронки и статусы |
| GET | `/api/leads` | Список сделок (`limit`, `page`, `query`) |
| GET | `/api/leads/{id}` | Одна сделка |
| GET | `/api/contacts` | Список контактов |
| GET | `/api/contacts/{id}` | Один контакт |
| GET | `/api/leads/{id}/notes` | Примечания к сделке (`limit`, `page`) |
| GET | `/api/catalogs` | Каталоги (товары) |
| POST | `/api/test-leads` | Создать тестовый лид (префикс `[BRATS-TEST]`, тег `brats_test`) |
| POST | `/api/test-lead-from-chat` | Создать контакт + тестовый лид из чата (телефон, имя, примечание, опционально email/channel); при заданном `DATABASE_URL` пишет в `conversations` и возвращает `conversation_id` для записи в календарь |
| GET | `/api/conversation/{id}` | Диалог по id + сообщения (для вкладки «Чат менеджера») |
| POST | `/api/conversation/{id}/messages` | Добавить сообщение в диалог (в т.ч. от имени менеджера: `role=assistant`) |
| PATCH | `/api/conversation/{id}/state` | Обновить состояние диалога (возврат из HANDOFF агенту) |
| POST | `/api/conversation/{id}/return-to-agent` | Вернуть диалог агенту (state + опционально системное сообщение-инструкция) |
| PATCH | `/api/leads/{id}/status` | Перевести сделку на этап воронки (тело: `{"status_id": <id>}`). Вызывается funnel-api при создании КП и при подготовке письма с договором |
| POST | `/api/leads/{id}/close` | Закрыть сделку (перевод в этап «Успешно реализовано» и т.п.) |

**Продвижение воронки при КП и договоре:** в config.env задайте этапы `AMOCRM_STATUS_KP_SENT` и `AMOCRM_STATUS_CONTRACT_SENT` (ID этапов из вашей воронки). **funnel-api** при успешном создании КП (`POST /api/proposals`) и при успешной подготовке письма с договором (`POST /api/contracts/prepare-email`) вызывает `PATCH /api/leads/{lead_id}/status` с соответствующим `status_id`. Для этого в config.env должен быть задан `AMOCRM_API_BASE_URL` (тот же, что для n8n). Сделка переводится только если у диалога есть `amocrm_lead_id` (для договора — из таблицы `conversations`).

Клиент в коде (`amocrm-api/client.py`) поддерживает также параметр `pipeline_ids` в `get_leads` для фильтра по воронкам.

### Создание лида из чата и запись в календарь

При передаче контакта в чате n8n вызывает `POST /api/test-lead-from-chat` с телом `{ "phone": "...", "name": "...", "note": "...", "email": "...", "channel": "website" }` (email и channel опциональны). Ответ:

- `lead_id`, `contact_id` — созданные сущности в AmoCRM;
- `conversation_id` — ID записи в PostgreSQL (`conversations`), если задан `DATABASE_URL` в config.env; иначе `null`.

Таблица **conversations** хранит слоты (имя, телефон, email) и связь с AmoCRM. Таблица **viewing_slots** (календарь показов) ссылается на `conversation_id` — по нему можно создавать слоты записи на просмотр для контакта. API календаря (свободные слоты 9–18, бронирование), КП и шаблоны договоров: **funnel-api** (порт 8011), см. [FUNNEL_CALENDAR_CP_CONTRACT.md](./FUNNEL_CALENDAR_CP_CONTRACT.md). Схема БД: `db-init/03_create_conversations.sql`, `db-init/07_viewing_slots_and_templates.sql`.

**Форма обратной связи на сайте:** в `site-integration/site-integration-example.html` при отправке формы всегда вызывается `POST .../api/test-lead-from-chat` (если в настройках не задан URL amocrm-api — используется fallback `origin + '/api/amocrm'`). Передаются `name`, `client_name`, `phone` (нормализованный, только цифры), `email`, `external_id` (visitor_id), `channel: 'site'`. Так лид и контакт создаются до первого сообщения в чате, слоты (client_name, phone, email) записываются в диалог. Чтобы агент не переспрашивал имя после заявки с формы, в n8n в системный промпт нужно подставлять слоты из ответа **Resolve** (см. шаг 2 в таблице выше и [n8n/BUSINESS_PROCESSES.md](../n8n/BUSINESS_PROCESSES.md), раздел 5).

## 4. Поля контакта в AmoCRM: что заполняется и как найти созданное при отладке

### Какие поля заполняются при создании из чата

| Где | Что заполняется | Условие |
|-----|-----------------|--------|
| **Контакт** | Имя (first_name / last_name) | Всегда из переданного `name`. |
| **Контакт** | Телефон | Заполняется, если найден ID поля: из **config.env** (`AMOCRM_CONTACT_PHONE_FIELD_ID`) или **автоматически** по данным AmoCRM (поля с code/name: PHONE, «телефон», «Source phone» и т.п.). |
| **Контакт** | Email | Заполняется, если передан в чате и найден ID поля: из **config.env** (`AMOCRM_CONTACT_EMAIL_FIELD_ID`) или **автоматически** по полям AmoCRM (EMAIL, «email», «почт», «Email раб» и т.п.). Если поле «Email раб» пустое — задайте в config.env **AMOCRM_CONTACT_EMAIL_FIELD_ID** (id поля из настроек контактов). |
| **Лид (сделка)** | Название | Всегда, с префиксом `[BRATS-TEST]` и именем. |
| **Лид** | Телефон (на карточке сделки) | Аналогично: **AMOCRM_LEAD_PHONE_FIELD_ID** или авто-подбор по custom_fields сделок. |
| **Лид** | Email (на карточке сделки) | **AMOCRM_LEAD_EMAIL_FIELD_ID** или авто-подбор по полям сделки. |
| **Лид** | Тег | `brats_test`. |
| **Лид** | Привязка контакта | Контакт связывается со сделкой. |
| **Лид** | Примечание | Текст из чата (note), если передан. |

**Автозаполнение полей:** при создании лида из чата amocrm-api запрашивает у AmoCRM списки доп. полей контактов и сделок и по **code**/name подбирает ID для телефона и email; приоритет у переменных config.env. Если в config.env заданы, например, `AMOCRM_CONTACT_PHONE_FIELD_ID=1275419` и `AMOCRM_CONTACT_EMAIL_FIELD_ID=1275421`, телефон и email из тела запроса `POST /api/test-lead-from-chat` записываются в контакт и при необходимости в сделку.

**Email нужно запрашивать и получать в процессе диалога.** Агент должен в подходящий момент (после телефона или при переходе к КП/договору) запросить email; когда клиент пришлёт его в сообщении, workflow n8n должен извлечь email из **исходного** сообщения и передать в `lead_payload` при вызове `POST /api/test-lead-from-chat` (поле `email`). Тогда amocrm-api запишет email в контакт и сделку по заданным в config.env ID полей. См. также [n8n/BUSINESS_PROCESSES.md](../n8n/BUSINESS_PROCESSES.md) (раздел 6, Maybe Create Lead — извлечение и передача email).

### Как узнать ID полей «Телефон» у контактов и у сделок в AmoCRM

1. **Настройки** (шестерёнка) → **Настройки воронок и этапов** или **Дополнительные поля** → выберите сущность **Контакты** или **Сделки**. В списке полей найдите нужное (например «Телефон», «Source phone», «Раб. тел.») — в API у поля есть числовой **id**.
2. Через API AmoCRM: `GET /api/v4/contacts/custom_fields` и `GET /api/v4/leads/custom_fields` (по токену доступа). В ответе у каждого поля — `id` и `name`. Для контакта подставьте id в **AMOCRM_CONTACT_PHONE_FIELD_ID**, для сделки — в **AMOCRM_LEAD_PHONE_FIELD_ID** в config.env.
3. **Выгрузка полей в файл (для справки по интеграции):** из корня репо выполните:
   ```bash
   python amocrm-api/scripts/export_custom_fields.py --out amocrm_fields.txt
   ```
   В `amocrm_fields.txt` сохранится таблица всех доп. полей сделок и контактов (id, name, type, code). Эту выгрузку можно хранить в репозитории как справочник по ID полей для текущего аккаунта AmoCRM. Подбор ID для телефона/email при создании лида из чата: `python amocrm-api/field_resolver.py` (выведет подобранные contact_phone_id, lead_phone_id и т.д.).
4. После изменения config.env перезапустите amocrm-api; новые лиды из чата будут создаваться уже с заполненным телефоном и email на контакте и на сделке.

### Как найти в AmoCRM последний лид или контакт, созданный при отладке

- **Сделки (лиды):**  
  - В разделе **Сделки** отфильтруйте по тегу **brats_test** или по названию, содержащему **`[BRATS-TEST]`**.  
  - Отсортируйте по дате создания (или «Сначала новые») — сверху будет последний созданный тестовый лид.
- **Контакты:**  
  - Лиды с тегом `brats_test` привязаны к контактам. Откройте нужную сделку → в карточке сделки отображаются связанные контакты — перейдите по ссылке на контакт.  
  - Либо в разделе **Контакты** отфильтруйте по дате создания и найдите по имени (например, «Клиент с сайта» или то имя, что вы вводили в чате).

Дополнительно: список последних лидов можно получить через API: `GET http://localhost:8010/api/leads?limit=10` — в ответе будут id и name; по имени с префиксом `[BRATS-TEST]` найдите нужный и откройте его в AmoCRM по id.

## 8. Сообщение менеджера в чат: из AmoCRM или отдельный интерфейс

После запроса клиента «связать с менеджером» (handoff) менеджер может отвечать клиенту в том же чате двумя способами.

### Вариант A: Прямо из AmoCRM (примечание → чат)

**Идея:** менеджер остаётся в карточке сделки в AmoCRM и пишет **примечание** к сделке; текст примечания попадает в наш чат и отображается клиенту как следующее сообщение от «консультанта».

**Что нужно:**

1. **Исходящий webhook AmoCRM** на событие «Добавление примечания» (сделка). Настраивается в амоМаркет → Web Hooks: URL вашего обработчика, событие — примечание к сделке (см. [формат webhooks AmoCRM](https://www.amocrm.ru/developers/content/crm_platform/webhooks-format)). Работа с WebHooks через API доступна на расширенном/профессиональном тарифе.
2. **Обработчик webhook** (n8n или amocrm-api): при получении события «примечание добавлено к сделке» взять `lead_id` (id сделки) и текст примечания; по `amocrm_lead_id` найти `conversation_id` в таблице `conversations`; вызвать `POST /api/conversation/{conversation_id}/messages` с `role=assistant` и текстом примечания. Тогда клиент увидит это сообщение в виджете чата при следующей загрузке истории (или при опросе/SSE, если реализовано).
3. **Соглашение:** примечания, которые должны уйти в чат, помечать отдельно (например, тип примечания или префикс в тексте), чтобы не дублировать в чат служебные заметки менеджера. Либо все примечания к сделке с определённым тегом/этапом считать «сообщением клиенту».

#### Нужен ли текущий OAuth-токен для приёма webhook?

**Нет.** Текущий токен (AMOCRM_ACCESS_TOKEN) используется только для **исходящих** запросов к API AmoCRM (создание лида, примечания, задачи и т.д.). При **приёме** webhook запрос идёт **от AmoCRM к вам**: AmoCRM шлёт POST на ваш URL. Токен при этом не участвует — AmoCRM не предъявляет ваш токен, вы просто принимаете POST.

**Что реально нужно для приёма webhook «примечание к сделке»:**

| Требование | Пояснение |
|------------|-----------|
| **Доступный из интернета URL** | AmoCRM должен иметь возможность отправить POST на ваш адрес. `localhost` не подойдёт: нужен сервер с белым IP или туннель (ngrok, cloudflared и т.п.), либо развёрнутый n8n/amocrm-api с публичным URL. |
| **Регистрация webhook в AmoCRM** | В интерфейсе: **амоМаркет** (или Настройки → Интеграции) → **Web Hooks** → указать URL и выбрать событие «Добавление примечания» для сущности «Сделка». Либо через API подписок (если тариф даёт такую возможность). |
| **Тариф AmoCRM** | Работа с WebHooks **через API** (создание/управление подписками программно) доступна на **расширенном или профессиональном** тарифе. Настройка webhook **вручную в интерфейсе** может быть доступна и на других тарифах — уточняйте в справке AmoCRM. |
| **Обработчик POST** | Узел Webhook в n8n или эндпоинт в amocrm-api, который принимает тело запроса, парсит `lead_id` и текст примечания, находит `conversation_id` по `amocrm_lead_id` в БД и добавляет сообщение в чат. Токен AmoCRM в обработчике нужен только если вы сами будете дергать AmoCRM API (например, прочитать сделку по id) — для простой записи в наш чат по данным из webhook токен не обязателен. |

**Итог:** для приёма webhook недостаточно «просто токена». Нужны: публичный URL обработчика, регистрация этого URL в AmoCRM как webhook на событие примечания к сделке и подходящий тариф. Токен по-прежнему нужен для ваших обращений к API AmoCRM (создание лида, примечаний и т.д.), но не для факта приёма webhook.

**Плюсы:** менеджер не переключается с AmoCRM, всё в одном окне.  
**Минусы:** нужна настройка webhook и тариф AmoCRM с WebHooks; примечания в AmoCRM не выглядят как диалог (нет пары «клиент — менеджер» в одном интерфейсе).

### Вариант B: Отдельная вкладка «Чат менеджера»

**Идея:** на нашем сайте/в админке открывается отдельная страница («вкладка»), где менеджер видит тот же диалог (по `conversation_id`, который можно взять из карточки сделки в AmoCRM или из списка переданных диалогов) и пишет сообщения. Сообщения сохраняются через `POST /api/conversation/{id}/messages` (role=assistant) и клиент видит их в том же чате.

**Что нужно:** страница с полем ввода и историей сообщений (GET диалога и сообщений — `GET /api/conversation/{id}` в amocrm-api), передача `conversation_id` в URL или выбор диалога из списка (например, диалоги в state=HANDOFF). Никаких webhook AmoCRM не требуется.

**Плюсы:** привычный вид чата, один поток переписки, не зависит от тарифа AmoCRM.  
**Минусы:** менеджер открывает вторую вкладку/интерфейс; нужно один раз узнать `conversation_id` по сделке (например, показывать его в карточке сделки в AmoCRM в кастомном поле или в примечании при handoff).

### Рекомендация

- Если важен **минимум переключений** и есть тариф с WebHooks — реализовать **вариант A** (примечание → webhook → сообщение в чат).
- Если нужна **простота без доработок AmoCRM** или нет WebHooks — сделать **вариант B** (отдельная вкладка с чатом по `conversation_id`). В карточке сделки при handoff можно сохранять в примечании ссылку на «Чат с клиентом: https://…?conversation_id=123».

В обоих вариантах клиент видит ответы менеджера в том же чате; возврат диалога агенту (чтобы бот дослал КП/договор) делается через `PATCH /api/conversation/{id}/state` или `POST /api/conversation/{id}/return-to-agent` (см. эндпоинты amocrm-api и раздел про handoff в [FUNNEL_CALENDAR_CP_CONTRACT.md](./FUNNEL_CALENDAR_CP_CONTRACT.md)).

### 8.1 Альтернатива n8n: оркестрация чата на FastAPI

Если нужно работать с **amocrm-api (8010)** и воронкой (КП, договор, слоты) **без n8n**, в качестве альтернативы можно использовать **отдельный FastAPI-сервис**, который повторяет логику текущего workflow: приём сообщения → память диалога → DLP → база знаний → LLM → создание лида при необходимости → ответ.

**Схема:**

```
Сайт / Telegram / ЛК  →  FastAPI (оркестратор)  →  amocrm-api (8010): resolve, test-lead-from-chat, messages, status
                              ↓                        funnel-api (8011): слоты, КП, договор
                         kb-service (8001): DLP, KB search
                              ↓
                         LLM (ProxyAPI / OpenAI)
```

**Что делает оркестратор (один или несколько эндпоинтов):**

| Шаг | Действие | Кого вызывает |
|-----|----------|----------------|
| 1 | Принять сообщение (`message`, `channel`, `external_id`, опционально `conversation_id`) | — |
| 2 | Разрешить диалог и получить историю + слоты (имя, телефон, email) | `GET {AMOCRM_API_BASE_URL}/api/conversation/resolve?channel=...&external_id=...` — ответ: `conversation_id`, `messages`, **`slots`**, `state`; слоты подставлять в системный промпт, чтобы агент не переспрашивал имя/контакты |
| 3 | Обезличить текст (DLP) | `POST kb-service:8001/api/dlp/sanitize-text` |
| 4 | Поиск в базе знаний | `POST kb-service:8001/api/kb/search` |
| 5 | Собрать промпт (воронка, слоты при необходимости с funnel-api, история), вызвать LLM | ProxyAPI / OpenAI |
| 6 | При наличии телефона в сообщении — создать лид/контакт | `POST amocrm-api:8010/api/test-lead-from-chat` |
| 7 | Записать реплику пользователя и ответ в диалог | `POST amocrm-api:8010/api/conversation/{id}/messages` (или через append в БД, если оркестратор имеет к ней доступ) |
| 8 | Вернуть ответ клиенту | — |

При интенте «КП» / «договор» оркестратор дополнительно дергает funnel-api (`POST /api/proposals`, `POST /api/contracts/prepare-email`) и при необходимости amocrm-api `PATCH /api/leads/{id}/status`.

**Где разместить:**

- **Вариант A:** новый микросервис в репозитории, например `chat-gateway/` или `sales-orchestrator/` — FastAPI-приложение с эндпоинтом `POST /webhook/chat` (или `/api/chat/message`), конфиг из `config.env` (URL amocrm-api, funnel-api, kb-service, OPENAI_API_KEY). Сайт и ЛК вместо n8n webhook вызывают этот URL.
- **Вариант B:** расширить существующий **sales-agent** (порт 8003), если он уже есть в проекте: добавить туда приём webhook-подобного запроса и цепочку вызовов amocrm-api, kb-service, LLM. Тогда amocrm-api остаётся только слоем доступа к AmoCRM и БД (conversations, LK); вся оркестрация — в sales-agent.

**Плюсы подхода без n8n:** один стек (Python), проще деплой и отладка, не нужен Node/n8n; полный контроль над логикой и промптом в коде.  
**Минусы:** сценарии (ветвления по интентам, создание КП/договора) нужно реализовывать в коде, а не в визуальном редакторе n8n.

Итог: **amocrm-api (8010)** можно использовать как единую точку доступа к AmoCRM и к диалогам (resolve, лиды, сообщения, смена статусов); оркестрацию чата при этом выполняет либо n8n, либо ваш FastAPI-сервис по описанной схеме.

## 9. Выводы о бизнес-процессах пользователя

На основе карты воронок, примеров сделок и примечаний (innovatoryclub):

- **Этапы сделки** задаются воронкой и статусом. Основной поток: квалификация (отдельная воронка или стартовые статусы) → показ/встреча → отправка КП/сметы/проекта → согласование проекта → ипотека (брокер, документы) → документы оформлены → оплата → успешно реализовано.
- **КП (коммерческое предложение):** факт отправки отражается переходом в статус «Отправили смету / КП / проект». Текст и файлы КП хранятся в примечаниях к сделке (типы common, вложения). В нашей системе КП ведутся в PostgreSQL (`proposals`) и синхронизируются в amoCRM примечанием и сменой статуса.
- **Ипотека:** этапы «Передано ипотечному брокеру» и «Подали документы на ипотеку»; детали — в примечаниях. В примечаниях по сделкам встречаются формулировки про ипотеку (например, «ипотека семейная», «пв почти 7 мил»).
- **Договор и продажа:** статусы «Документы оформлены», «Поступила полная оплата»; финалы в воронке «Работы маркетинг» — «Успешно реализовано» (142), «Закрыто и не реализовано» (143). Договор может храниться в примечаниях (файл) или во внешней системе; в amoCRM фиксируется переход в статус оплаты/закрытия.
- **Каталог товаров** в amoCRM отсутствует — состав КП и договоров формируем из таблицы `products` в PostgreSQL.

Согласование с целевыми процессами — [BUSINESS_PROCESSES.md](./BUSINESS_PROCESSES.md).
