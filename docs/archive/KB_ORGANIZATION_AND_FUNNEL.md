# Организация базы знаний и ведение клиента по воронке продаж

Документ фиксирует принципы организации KB, принятые в проекте, описание скриптов загрузки и их аналогов в админке, а также то, как предполагается вести клиента по воронке продаж в соответствии с бизнес-процессами проекта.

**Связанные документы:** [BUSINESS_PROCESSES.md](../BUSINESS_PROCESSES.md), [docs/AMOCRM.md](./AMOCRM.md), [docs/KB_MEDIA_STRUCTURE.md](./KB_MEDIA_STRUCTURE.md).

---

## 1. Принципы организации KB

### 1.1. Единая таблица и метаданные

- **Хранение:** PostgreSQL, таблица `knowledge_base` (pgvector). Текст — в поле `content`, вектор — в `embedding`, всё остальное — в **`metadata`** (JSONB).
- **Схему БД не меняем** под новые типы контента: картинки и документы хранятся в `metadata.images` и `metadata.documents`, а не в отдельных таблицах.
- **Категории** (product_info, sales_script, objection_handling, contacts, pricing, location, tone_of_voice, target_audience) и **целевая аудитория** (end_buyer, realtor, both), **приоритет**, **источник** (`source`) задаются в metadata и используются при поиске и фильтрации.

### 1.2. Мультимодальность: картинки и документы в ответе агента

- **`metadata.images`** — массив объектов `{ url, description?, alt? }`. URL может вести на сайт, Yandex.Disk или на ваш сервер (статическая папка / API). Картинки с диска раздаются по HTTP, в KB хранится только URL.
- **`metadata.documents`** — массив объектов `{ url, title?, description? }` для PDF, DOCX и других файлов.
- **Поиск** (`POST /api/kb/search`) возвращает полный `metadata`. Цепочка n8n собирает из результатов релевантные медиа и добавляет в ответ агента поле **`media`** (`images`, `documents`). Чат на сайте отображает их под текстом ответа.
- **Промпт агента** предписывает: при наличии в контексте «Релевантные медиа» — кратко упомянуть приложенные фото/документы, длинные URL в текст не вставлять.

Подробнее: [docs/KB_MEDIA_STRUCTURE.md](./KB_MEDIA_STRUCTURE.md), [docs/CHAT_SHOW_IMAGES.md](./CHAT_SHOW_IMAGES.md).

### 1.3. Источники контента и дедупликация

- Контент попадает в KB: (1) скриптами из папки `data/`, (2) импортом по URL через мультимодального библиотекаря (`/api/mmkb/import_from_url`, `/api/mmkb/save`), (3) ручным добавлением и редактированием в KB Admin.
- Библиотекарь проверяет релевантность и дубликаты (в т.ч. по `metadata.source_url`) перед сохранением. При импорте из текста/файлов URL картинок и документов **извлекаются автоматически** и попадают в metadata — запросы по фото/ссылкам релевантно возвращают медиа.

### 1.4. Идентификация посетителя и отсутствие дубликатов лидов

- Сайт передаёт в n8n **`external_id`** (например, из cookie или Yandex Metrika client ID), чтобы один и тот же посетитель распознавался при возврате.
- amocrm-api при создании лида из чата принимает **`external_id`** и **`channel`**; перед созданием ищет диалог по ним. Если у диалога уже есть **`amocrm_lead_id`**, новый лид не создаётся, возвращается существующий. Клиенту не нужно повторно вводить телефон или ФИО для избежания дубликата лида.

## 2. Скрипты загрузки KB и аналоги в админке

### 2.1. Скрипты (из корня проекта)

| Скрипт | Назначение | Когда использовать |
|--------|------------|---------------------|
| **`scripts/load_kb_from_data.py`** | Загрузка БЗ из папки `data/` в KB через API (без очистки таблицы). | Добавить/обновить контент из файлов; после изменения файлов в `data/`. |
| **`scripts/reset_and_load_kb.py`** | Очистка таблицы `knowledge_base` (TRUNCATE) + запуск `load_kb_from_data.py`. | Полная перезаливка БЗ «с нуля». |
| **`scripts/check_kb_relevance.py`** | Проверка релевантности: вопросы из `data/БЗ/вопросы_neurocrm.txt` (или указанного файла) отправляются в `/api/kb/search`, выводятся топ результатов и схожесть. | Оценка качества наполнения KB и подбора чанков под типовые запросы. |

**Общие параметры (load_kb_from_data / reset_and_load_kb):**

- `--data-dir` — корневая папка с данными (по умолчанию `data`).
- `--chunk-size`, `--overlap` — размер чанка и перекрытие при разбиении текста.
- `--encoding` — кодировка для .txt (по умолчанию utf-8, fallback cp1251).
- `--dry-run` — только список файлов, без отправки в API (у reset_and_load_kb — без очистки и без загрузки).
- `--include-test-questions` — включить в загрузку файл с тестовыми вопросами (по умолчанию он исключён и используется только для проверки релевантности).

**Поведение по типам файлов:**

- **.txt, .md:** текст читается локально, из него извлекаются URL картинок и документов; в запрос `/api/kb/import` передаётся `content` и при наличии — `metadata: { images, documents }`.
- **.docx, .pdf, .pptx:** файл отправляется в **`POST /api/kb/import_document`**; kb-service извлекает текст, из текста — URL медиа, и при создании чанков подставляет их в metadata.

Категория и целевая аудитория определяются по имени файла и пути (маппинг в скрипте: price/прайс → pricing, adress/адрес → location, photo_descriptions/Детально_ → product_info, presentation_realtor → product_info + realtor и т.д.).

**Примеры:**

```bash
cd /projects/brats
source config.env   # или export KB_API_URL=http://localhost:8001

# Только список файлов
python scripts/load_kb_from_data.py --data-dir data --dry-run

# Загрузить без очистки
python scripts/load_kb_from_data.py --data-dir data

# Полная перезаливка (очистка + загрузка)
python scripts/reset_and_load_kb.py --data-dir data

# Проверка релевантности по вопросам из файла
python scripts/check_kb_relevance.py
```

**Требования:** в `config.env` задан `KB_API_URL` (по умолчанию http://localhost:8001); для `reset_and_load_kb` — также `DATABASE_URL` (та же БД, что у kb-service). kb-service должен быть запущен при загрузке.

### 2.2. Аналоги в админке KB (Streamlit, порт 8501)

| Действие | В скриптах | В админке KB |
|----------|------------|--------------|
| Загрузить текст в KB | `load_kb_from_data.py` (файлы .txt/.md) или вызов API `/api/kb/import` с `content` и опционально `metadata` | Раздел импорта: ввод текста, выбор категории/аудитории, при необходимости — передача `metadata` (images/documents) через вызов `kb_import_text(..., metadata=...)`. |
| Загрузить документ (PDF, DOCX, PPTX) | `load_kb_from_data.py` вызывает `/api/kb/import_document`; сервис сам извлекает текст и URL медиа | Загрузка файла через UI (если реализовано) или через API вручную; на стороне API — тот же импорт с автоизвлечением URL в metadata. |
| Очистить всю KB и загрузить заново | `reset_and_load_kb.py` | Нет прямого аналога; очистка — через прямой доступ к БД или отдельный скрипт/команду. |
| Просмотр и поиск по чанкам | — | «Обзор и поиск»: запрос к `/api/kb/search`, список результатов с категорией, источником, превью контента. |
| Редактирование чанка (текст, категория, приоритет, теги, источник) | — | «Редактирование chunk»: форма с полями content, category, target_audience, priority, tags, source; кнопка «Сохранить» → `PUT /api/kb/{id}`. |
| Редактирование медиа (картинки, документы) | — | В форме редактирования чанка — блок «Медиа (картинки и документы)»: просмотр текущих `metadata.images` и `metadata.documents`, добавление новых (URL + подпись/название), удаление. При сохранении в payload уходит `metadata: { images, documents }`. |
| Импорт по URL (сайт заказчика) + библиотекарь | — | Мультимодальная KB: импорт по URL, просмотр решения библиотекаря, сохранение в KB с изображениями и метаданными (см. BUSINESS_PROCESSES.md, раздел 10). |

Итог: скрипты обеспечивают массовую загрузку из `data/` с автоизвлечением медиа-URL; админка — точечное редактирование, поиск и при необходимости ручной импорт текста/файлов с возможностью задать медиа.

## 3. Ведение клиента по воронке продаж

Ниже — как предполагается вести клиента по воронке в рамках текущих бизнес-процессов проекта (источники: BUSINESS_PROCESSES.md, docs/AMOCRM.md).

### 3.1. Общий поток: от лида до договора

1. **Появление лида** — сайт (чат/форма + трекинг), Telegram, Avito или другие каналы. Сообщения приходят в n8n (webhook sales-agent-kb).
2. **Нормализация и память диалога** — n8n нормализует ввод, при наличии `external_id` и `channel` запрашивает у amocrm-api разрешение диалога (`/api/conversation/resolve`); подтягивается история переписки и при необходимости — существующий лид (без создания дубликата).
3. **Обезличивание (DLP)** — текст сообщения перед передачей в LLM обезличивается (kb-service DLP).
4. **Поиск в KB и формирование контекста** — запрос в `/api/kb/search` по сообщению и контексту; из результатов собирается текстовый контекст и релевантные медиа (images/documents). Контекст и слоты календаря (свободные окна 9–18) передаются в промпт нейропродажника.
5. **Ответ агента** — LLM формирует ответ по базе знаний, истории и слотам; в ответ клиенту добавляется `media` (картинки/документы). При запросе «показать фото отделки» или «дать ссылку на презентацию» агент возвращает релевантные ссылки/медиа из KB.
6. **Создание лида** — при появлении в сообщении телефона n8n вызывает amocrm-api `POST /api/test-lead-from-chat` с телефоном, именем, примечанием, `external_id`, `channel`. Если по `external_id`+`channel` уже есть диалог с лидом, новый лид не создаётся. Иначе создаётся контакт и сделка в AmoCRM, при наличии DATABASE_URL — запись в `conversations` и возврат `conversation_id` для календаря показов.
7. **Запись на просмотр** — при желании клиента записаться на просмотр агент предлагает свободные слоты (из funnel-api или аналога); выбор слота может сохраняться в conversation/slots.
8. **Дальнейшие этапы воронки** (по BUSINESS_PROCESSES.md и AmoCRM): квалификация → показ → КП/смета → согласование проекта → ипотека (брокер, документы) → документы оформлены → оплата → успешно реализовано. КП и договоры в целевой архитектуре ведутся в PostgreSQL и синхронизируются с AmoCRM (примечания, статусы сделки).

### 3.2. Роль KB в воронке

- **До контакта (аноним):** агент отвечает по KB, привлекает оставить контакт, может показывать релевантные фото и документы из metadata.
- **После контакта:** агент ведёт по воронке — запись на просмотр (слоты), ссылки на сайт/отделку, КП, договор, ипотека — опираясь на те же чанки (product_info, pricing, objection_handling и т.д.) и память диалога. Человеку передаётся по запросу клиента (HUMAN IN THE LOOP); лид не дублируется при повторном заходе с тем же external_id.
- **Обогащение KB** (BUSINESS_PROCESSES.md, раздел 9): вопросы и возражения из диалогов и примечаний менеджеров могут использоваться для добавления новых чанков или обновления приоритетов, чтобы ответы агента лучше соответствовали реальным запросам.

### 3.3. FSM нейропродажника (целевые состояния)

По BUSINESS_PROCESSES.md: приветствие (GREETING) → квалификация (QUALIFYING) → при возражениях — обработка (OBJECTIONS) или передача менеджеру (HANDOFF); при готовности к КП — предложение (PROPOSAL) → согласование (NEGOTIATING) → финализация КП (FINALIZED) → договор (CONTRACT) → завершение (COMPLETED). Текущий workflow n8n (sales-agent-kb) обеспечивает один шаг ответа с KB и лидом; детальный FSM и переходы между этапами (КП, договор) реализуются в связке с proposal-generator и funnel-api согласно тем же бизнес-процессам.

## 4. Краткая сводка

| Тема | Решение |
|------|---------|
| Структура KB | Одна таблица `knowledge_base`, медиа в `metadata.images` и `metadata.documents`. |
| Релевантные картинки/документы в ответе | Поиск возвращает metadata; n8n собирает медиа и отдаёт в `response.media`; чат отображает их. |
| Загрузка из data/ | `load_kb_from_data.py` (и при необходимости `reset_and_load_kb.py`); URL из текста/файлов автоматически попадают в metadata. |
| Редактирование и ручной ввод | KB Admin: поиск, редактирование чанков, в т.ч. блок «Медиа» (images/documents), сохранение через API. |
| Воронка и лиды | Агент ведёт по этапам (квалификация → показ → КП → договор); лид создаётся при телефоне; дубликат не создаётся при том же external_id+channel. |
| Идентификация посетителя | Сайт передаёт stable `external_id` (cookie / Yandex ID); один посетитель = один диалог и один лид после контакта. |
