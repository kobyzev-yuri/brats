# Краткая инструкция: Дедупликация при записи в KB

## Правило для Gemini

```
При записи смысловых блоков в Knowledge Base:

1. ПЕРЕД ЗАПИСЬЮ:
   - Сгенерируй идентификатор блока
   - Проверь существование в KB
   - Если точный дубликат → пропусти запись
   - Если семантически похожий (>90%) → пропусти запись

2. ИДЕНТИФИКАТОРЫ:
   - Заголовок: "title_{tab}_{type}"
   - Описание: "description_{tab}_{hash}"
   - Что включает: "includes_{tab}_{hash}"
   - Стоимость: "price_{tab}_{normalized_price}"
   - Контакты: "contacts_{hash}" (один раз для всех)
   - Преимущества: "advantages_{hash}" (один раз для всех)

3. СПЕЦИАЛЬНЫЕ СЛУЧАИ:
   - Контакты/преимущества одинаковы для всех вкладок → записать один раз
   - Изменение цены → это обновление, не дубликат

4. ЛОГИРОВАНИЕ:
   - "✅ Запись нового блока"
   - "⏭️ Пропуск дубликата"
   - "🔄 Обновление блока"
```

## Алгоритм

```
Для каждого блока:
1. Генерируй ID
2. Проверь существование в KB
3. Если существует:
   - Точный дубликат? → Пропустить
   - Семантически похожий (>90%)? → Пропустить
   - Обновление? → Обновить
4. Если не существует → Записать
```

## Чек-лист

- [ ] Сгенерирован ID блока
- [ ] Проверено существование в KB
- [ ] Определен тип (новый/дубликат/обновление)
- [ ] Принято решение о записи
- [ ] Записано в KB (если нужно)
- [ ] Зафиксировано в логе

## Важно

**Не записывай повторно те же самые блоки!**
Всегда проверяй на дубликаты перед записью.
