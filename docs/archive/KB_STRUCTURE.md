# Структура базы знаний для агента продаж

## Обзор

Данный документ описывает, как структурировать информацию из `kb_info.txt` для эффективного использования в Knowledge Base (KB) агента продаж через RAG (Retrieval Augmented Generation).

## Цель базы знаний

База знаний должна обеспечивать агента продаж:
1. **Точной информацией о продукте** (дома, цены, характеристики)
2. **Скриптами продаж** для разных типов клиентов
3. **Обработкой возражений** с готовыми ответами
4. **Информацией о целевой аудитории** и триггерах покупки
5. **Tone of Voice** для поддержания единого стиля общения
6. **Контактами и процедурами** (показы, график работы)

---

## Рекомендуемая структура метаданных

### Хранение в PostgreSQL с pgvector

База знаний хранится в **PostgreSQL** с использованием расширения **pgvector** для векторного поиска:

1. **Таблица `knowledge_base`** — основная таблица для хранения chunks базы знаний
2. **Векторные embeddings** — генерируются через OpenAI API (text-embedding-3-small) и хранятся в поле `embedding vector(1536)`
3. **Метаданные в JSONB** — гибкая структура для фильтрации и контекста
4. **Индексы** — IVFFlat индекс для быстрого векторного поиска, GIN индексы для метаданных

**Преимущества PostgreSQL (pgvector):**
- Единая база данных для всех данных (KB, объекты, диалоги)
- Эффективный векторный поиск через pgvector
- Полный контроль над схемой и индексами
- Масштабируемость для больших объемов данных
- Интеграция с каталогом объектов недвижимости

**Схема таблицы:**
```sql
CREATE TABLE knowledge_base (
    id SERIAL PRIMARY KEY,
    content TEXT NOT NULL,
    embedding vector(1536),
    metadata JSONB NOT NULL,
    version VARCHAR(20) DEFAULT '1.0',
    last_updated TIMESTAMP DEFAULT NOW(),
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX ON knowledge_base USING ivfflat (embedding vector_cosine_ops);
CREATE INDEX ON knowledge_base USING GIN (metadata);
```

  "context": {
    "use_case": "greeting",
    "stage": "early"
  }

```

**Контент:**
```
Tone of Voice: «Аспиративный тёплый» (база бренда)

Как звучит: уверенно‑спокойно, дружелюбно, с лёгкой ноткой lifestyle.

## Рекомендации по chunking (разбиению на части)

### Принципы chunking

1. **Семантическая целостность**: каждый chunk должен содержать законченную мысль или информацию
2. **Размер chunks**: 100-500 токенов (примерно 150-750 слов) для оптимального баланса между контекстом и точностью поиска
3. **Перекрытие**: минимальное перекрытие между chunks (10-20%) для сохранения контекста
4. **Метаданные**: каждый chunk должен иметь полные метаданные для фильтрации

### Примеры хорошего chunking

✅ **Хорошо:**
```
Chunk 1: "Черновая отделка — 8 350 000 ₽. Подробнее: [ссылка]"
Chunk 2: "Предчистовая отделка — 8 950 000 ₽. Подробнее: [ссылка]"
```

❌ **Плохо:**
```
Chunk: "Черновая отделка — 8 350 000 ₽. Предчистовая отделка — 8 950 000 ₽. Стандартный ремонт — 9 650 000 ₽. Дизайнерский ремонт — 10 450 000 ₽."
```
(слишком много информации в одном chunk, сложнее найти конкретную цену)

## Использование в RAG через PostgreSQL (pgvector)

### Процесс поиска в KB

#### 1. Генерация embeddings для запроса

Перед поиском необходимо сгенерировать векторное представление запроса клиента:

```python
from openai import OpenAI

openai_client = OpenAI()

def generate_embedding(text: str) -> list:
    """Генерация embedding для текста через OpenAI"""
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding
```

#### 2. Векторный поиск в PostgreSQL

Поиск выполняется через pgvector с использованием cosine similarity:

```python
import asyncpg

async def search_knowledge_base(
    query: str,
    target_audience: str = "both",
    category: str = None,
    limit: int = 5,
    similarity_threshold: float = 0.6
):
    """
    Поиск в базе знаний через векторный поиск в PostgreSQL.
    
    Args:
        query: Текстовый запрос клиента
        target_audience: Целевая аудитория ('end_buyer', 'realtor', 'both')
        category: Категория для фильтрации (опционально)
        limit: Количество результатов
        similarity_threshold: Минимальный порог схожести (0-1)
    
    Returns:
        Список релевантных chunks с similarity scores
    """
    # Генерация embedding для запроса
    query_embedding = generate_embedding(query)
    
    # Построение SQL запроса с фильтрацией
    where_clauses = ["is_active = TRUE"]
    params = [query_embedding]
    param_idx = 2
    
    if target_audience != "both":
        where_clauses.append(f"metadata->>'target_audience' = ${param_idx}")
        params.append(target_audience)
        param_idx += 1
    
    if category:
        where_clauses.append(f"metadata->>'category' = ${param_idx}")
        params.append(category)
        param_idx += 1
    
    where_sql = " AND ".join(where_clauses)
    
    # Выполнение запроса
    async with asyncpg.create_pool("postgresql://...") as pool:
        async with pool.acquire() as conn:
            results = await conn.fetch(f"""
                SELECT 
                    id,
                    content,
                    metadata,
                    1 - (embedding <=> $1::vector) as similarity
                FROM knowledge_base
                WHERE {where_sql}
                  AND 1 - (embedding <=> $1::vector) > ${param_idx}
                ORDER BY embedding <=> $1::vector
                LIMIT ${param_idx + 1}
            """, *params, similarity_threshold, limit)
            
            return [
                {
                    "id": r["id"],
                    "content": r["content"],
                    "metadata": r["metadata"],
                    "similarity": float(r["similarity"])
                }
                for r in results
            ]
```

#### 3. Использование в sales-agent

Интеграция поиска KB в агента продаж:

```python
# sales-agent/llm/rag.py
async def get_kb_context(
    user_query: str,
    conversation_context: dict
) -> str:
    """
    Получение релевантного контекста из базы знаний для ответа агента.
    """
    # Определение целевой аудитории из контекста
    target_audience = conversation_context.get("target_audience", "both")
    
    # Поиск в KB
    kb_results = await search_knowledge_base(
        query=user_query,
        target_audience=target_audience,
        limit=5
    )
    
    # Формирование контекста для LLM
    context_parts = []
    for result in kb_results:
        context_parts.append(f"[KB] {result['content']}")
    
    return "\n\n".join(context_parts)
```

#### 4. Оптимизация производительности

- **IVFFlat индекс**: для быстрого приближенного поиска (рекомендуется для больших объемов)
- **GIN индексы**: для фильтрации по метаданным (JSONB)
- **Кэширование**: часто используемые запросы можно кэшировать в Redis

### Приоритизация результатов поиска

1. **По similarity**: выше similarity = более релевантный результат
2. **По priority**: `high` > `medium` > `low`
3. **По target_audience**: соответствие типу клиента
4. **По use_case и stage**: соответствие текущему этапу воронки

## Структура для импорта данных в PostgreSQL

### Формат данных для импорта

Данные из `kb_import_example.py` экспортируются в JSON формат, который затем загружается в PostgreSQL:

```json
{
  "chunks": [
    {
      "content": "Черновая отделка — 8 350 000 ₽\nПодробнее: https://innovatory-club.ru/katalog#!/tab/1063728081-1",
      "metadata": {
        "category": "product_info",
        "subcategory": "pricing",
        "target_audience": "end_buyer",
        "priority": "high",
        "tags": ["цена", "отделка", "ремонт"],
        "source": "kb_info.txt",
        "version": "1.0",
        "related_links": ["https://innovatory-club.ru/katalog#!/tab/1063728081-1"],
        "context": {
          "use_case": "proposal",
          "stage": "middle"
        }
      }
    }
  ]
}
```

### Процесс загрузки в PostgreSQL

1. **Подготовка данных**
   - Используйте скрипт `kb_import_example.py` для генерации chunks
   - Скрипт экспортирует chunks в JSON формат

2. **Генерация embeddings**
   - Для каждого chunk генерируется embedding через OpenAI API
   - Используется модель `text-embedding-3-small` (1536 измерений)

3. **Загрузка в PostgreSQL**
   ```python
   import asyncpg
   import json
   from openai import OpenAI
   
   async def import_kb_chunks(json_file: str, db_url: str):
       """Импорт chunks из JSON в PostgreSQL"""
       openai_client = OpenAI()
       
       with open(json_file, 'r', encoding='utf-8') as f:
           data = json.load(f)
       
       async with asyncpg.create_pool(db_url) as pool:
           async with pool.acquire() as conn:
               for chunk in data['chunks']:
                   # Генерация embedding
                   embedding = openai_client.embeddings.create(
                       model="text-embedding-3-small",
                       input=chunk['content']
                   ).data[0].embedding
                   
                   # Вставка в БД
                   await conn.execute("""
                       INSERT INTO knowledge_base (content, embedding, metadata, version)
                       VALUES ($1, $2::vector, $3, $4)
                   """, 
                       chunk['content'],
                       embedding,
                       json.dumps(chunk['metadata']),
                       chunk['metadata'].get('version', '1.0')
                   )
   ```

4. **Обновление индексов**
   - После загрузки данных индексы автоматически обновляются
   - Для больших объемов можно пересоздать IVFFlat индекс с оптимальными параметрами

## Рекомендации по обновлению KB

1. **Версионирование**: используйте поле `version` в metadata для отслеживания изменений
2. **Дата обновления**: поле `last_updated` для понимания актуальности информации
3. **Источник**: поле `source` для отслеживания происхождения данных
4. **Регулярные обновления**: обновляйте цены, наличие объектов, контакты
5. **A/B тестирование**: можно создавать разные версии chunks для тестирования эффективности
