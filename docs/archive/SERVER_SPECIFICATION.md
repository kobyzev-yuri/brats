# Спецификация сервера для деплоймента системы BRATS

## Общая информация

**Назначение:** Развертывание мультиагентной системы нейроаналитики и нейропродаж для застройщика коттеджного посёлка.

**Ожидаемая нагрузка:** Средняя/низкая (не предполагается высокой нагрузки на старте)

**Требования к расширяемости:** Система должна иметь возможность масштабирования при росте нагрузки

**Сетевая архитектура:** Локальных сеток не предполагается, используется внешний прокси (proxyapi.ru или аналог)

---

## Операционная система

**Дистрибутив:** Ubuntu 24.04 LTS (Server Edition)

**Обоснование:**
- Долгосрочная поддержка до 2029 года
- Стабильность и безопасность
- Широкая поддержка современного ПО
- Оптимизация для серверных задач

**Минимальные требования к установке:**
- Ubuntu Server 24.04 LTS (64-bit)
- Полная установка (не минимальная) для удобства администрирования

---

## Аппаратная спецификация

### Вариант 1: Базовый (для старта)

**Процессор (CPU):**
- **Минимум:** 4 ядра (4 vCPU)
- **Рекомендуется:** 6-8 ядер (6-8 vCPU)
- **Архитектура:** x86_64 (AMD64)
- **Частота:** 2.4 GHz и выше

**Оперативная память (RAM):**
- **Минимум:** 16 GB
- **Рекомендуется:** 32 GB
- **Тип:** DDR4/DDR5 ECC (предпочтительно для серверов)

**Дисковое пространство:**
- **SSD:** 200 GB (система, приложения, Docker)
- **Дополнительно:** 500 GB SSD для базы данных PostgreSQL
- **Итого:** 700 GB SSD
- **Тип:** NVMe SSD (предпочтительно) или SATA SSD
- **Файловая система:** ext4 или xfs

**Сетевая карта:**
- **Скорость:** 1 Gbps (минимум)
- **Рекомендуется:** 10 Gbps (для будущего масштабирования)

**Резервное копирование:**
- Отдельный диск/том для бэкапов: 500 GB (опционально, можно использовать внешнее хранилище)

### Вариант 2: С запасом для расширения

**Процессор (CPU):**
- **Рекомендуется:** 8-12 ядер (8-12 vCPU)
- **Архитектура:** x86_64 (AMD64)
- **Частота:** 2.6 GHz и выше

**Оперативная память (RAM):**
- **Рекомендуется:** 64 GB
- **Тип:** DDR4/DDR5 ECC

**Дисковое пространство:**
- **SSD (система):** 250 GB NVMe SSD
- **SSD (база данных):** 1 TB NVMe SSD
- **SSD (логи, кэш):** 500 GB NVMe SSD
- **Итого:** 1.75 TB SSD
- **Файловая система:** xfs (для больших объемов данных)

**Сетевая карта:**
- **Скорость:** 10 Gbps

**Резервное копирование:**
- Отдельный том для бэкапов: 1 TB

## Программное обеспечение

### Базовое ПО

1. **Операционная система:**
   - Ubuntu 24.04 LTS Server
   - Обновления безопасности включены автоматически

2. **Веб-сервер:**
   - **Nginx 1.26+** (reverse proxy, SSL termination)
   - SSL/TLS сертификаты (Let's Encrypt через certbot)

3. **Контейнеризация:**
   - **Docker 24.0+** (или новее)
   - **Docker Compose 2.24+**

4. **Системные утилиты:**
   - `curl`, `wget`, `git`, `vim`, `htop`, `iotop`, `netstat`, `ss`
   - `ufw` (Uncomplicated Firewall)
   - `fail2ban` (защита от брутфорса)
   - `logrotate` (ротация логов)

### Основные сервисы системы

1. **База данных:**
   - **PostgreSQL 15+** (или 16+)
   - Расширение **pgvector** для векторного поиска
   - Рекомендуемая конфигурация:
     - `shared_buffers`: 25% от RAM
     - `effective_cache_size`: 50-75% от RAM
     - `maintenance_work_mem`: 2-4 GB
     - `work_mem`: 256-512 MB
     - `max_connections`: 200-300

2. **Кэш и очереди:**
   - **Redis 7+** (или 7.2+)
   - Конфигурация:
     - `maxmemory`: 4-8 GB (для варианта 1) или 8-16 GB (для варианта 2)
     - `maxmemory-policy`: allkeys-lru

3. **Оркестратор:**
   - **n8n** (self-hosted, Docker)
   - Порт: 5678 (внутренний, через Nginx)

4. **Python сервисы (FastAPI):**
   - **Python 3.11+** (или 3.12+)
   - **kb-service** (API для базы знаний)
   - **sales-analytic** (сервис аналитики)
   - **sales-agent** (нейропродажник)
   - **proposal-generator** (генератор КП)

5. **Внешние зависимости:**
   - API GigaChat / YandexGPT (внешние, через proxyapi.ru)
   - API amoCRM (внешний)
   - API Яндекс.Метрика (внешний)

## Сетевая архитектура

### Топология

```
Интернет
    ↓
[proxyapi.ru или аналог] ← Внешний прокси-сервис
    ↓
[Сервер BRATS]
    ├── Nginx (порт 80/443) ← Reverse proxy
    │   ├── → n8n (внутренний порт 5678)
    │   ├── → kb-service (внутренний порт 8001)
    │   ├── → sales-analytic (внутренний порт 8002)
    │   ├── → sales-agent (внутренний порт 8000)
    │   └── → proposal-generator (внутренний порт 8003)
    │
    ├── PostgreSQL (порт 5432) ← Только локальный доступ
    └── Redis (порт 6379) ← Только локальный доступ
```

### Настройка Nginx

**Основные функции:**
- Reverse proxy для всех внутренних сервисов
- SSL/TLS termination
- Rate limiting
- Логирование запросов
- Health checks

**Примерная конфигурация:**
- Все сервисы доступны только через Nginx
- Внутренние порты не открыты наружу
- SSL сертификаты от Let's Encrypt

### Firewall (UFW)

**Открытые порты:**
- `22/tcp` - SSH (ограничить доступ по IP)
- `80/tcp` - HTTP (для Let's Encrypt)
- `443/tcp` - HTTPS

**Закрытые порты (только localhost):**
- `5432/tcp` - PostgreSQL
- `6379/tcp` - Redis
- `5678/tcp` - n8n (внутренний)
- `8000-8003/tcp` - FastAPI сервисы (внутренние)

## Мониторинг и логирование

### Системный мониторинг

1. **Мониторинг ресурсов:**
   - `htop` / `top` - CPU, RAM
   - `iotop` - дисковые операции
   - `nethogs` - сетевая активность
   - `df -h` - использование дисков

2. **Мониторинг сервисов:**
   - Docker контейнеры: `docker stats`
   - PostgreSQL: встроенные метрики
   - Redis: `INFO` команда

3. **Логирование:**
   - Системные логи: `/var/log/syslog`
   - Nginx логи: `/var/log/nginx/`
   - Docker логи: `docker logs`
   - Логи приложений: централизованное хранилище (опционально)

### Рекомендуемые инструменты (опционально)

- **Prometheus + Grafana** (для расширенного мониторинга)
- **ELK Stack** (для централизованного логирования)
- **Uptime Kuma** (мониторинг доступности)

## Резервное копирование

### Стратегия бэкапов

1. **База данных PostgreSQL:**
   - Ежедневные полные бэкапы
   - Еженедельные архивы WAL
   - Хранение: минимум 30 дней

2. **Конфигурации:**
   - Docker Compose файлы
   - Nginx конфигурации
   - Environment файлы (.env)
   - n8n workflows

3. **Данные приложений:**
   - n8n данные (volumes)
   - Логи (если критичны)

### Инструменты

- `pg_dump` / `pg_dumpall` для PostgreSQL
- `rsync` для файлов
- Автоматизация через cron

## Безопасность

### Базовые меры

1. **Обновления:**
   - Автоматические обновления безопасности
   - Регулярные обновления пакетов

2. **Доступ:**
   - SSH только по ключам (отключить парольную аутентификацию)
   - Ограничение доступа по IP (если возможно)
   - Отключение root логина

3. **Firewall:**
   - UFW настроен и активен
   - Минимально необходимые порты открыты

4. **Защита от атак:**
   - fail2ban для SSH и веб-сервисов
   - Rate limiting в Nginx
   - DDoS защита через proxyapi.ru

5. **Сертификаты:**
   - SSL/TLS для всех веб-сервисов
   - Автоматическое обновление Let's Encrypt

## Масштабирование

### Горизонтальное масштабирование

При росте нагрузки возможны следующие варианты:

1. **Разделение сервисов:**
   - Отдельный сервер для PostgreSQL
   - Отдельный сервер для Redis
   - Отдельные серверы для FastAPI сервисов

2. **Балансировка нагрузки:**
   - Несколько инстансов FastAPI сервисов
   - Load balancer перед сервисами

3. **Кэширование:**
   - Увеличение памяти Redis
   - CDN для статики (если появится)

### Вертикальное масштабирование

- Увеличение CPU и RAM на текущем сервере
- Улучшение дисков (NVMe, больше IOPS)
- Увеличение пропускной способности сети
