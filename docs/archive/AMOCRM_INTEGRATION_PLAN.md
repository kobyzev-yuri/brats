# План интеграции amoCRM: n8n + FastAPI

Документ связывает бизнес-процессы из [BUSINESS_PROCESSES.md](./BUSINESS_PROCESSES.md) с интеграцией amoCRM на двух уровнях: **n8n** (существующий оркестратор) и **независимый FastAPI** (прямые вызовы API для сервисов и скриптов). Токен уже работает в одном из n8n workflow — те же учётные данные можно использовать в FastAPI через `.env`.

---

## 1. Какие бизнес-процессы задействуют amoCRM

По [BUSINESS_PROCESSES.md](./BUSINESS_PROCESSES.md) amoCRM участвует в следующем.

| Процесс | Раздел BP | Чтение amoCRM | Запись amoCRM | Кто использует |
|--------|-----------|----------------|---------------|-----------------|
| Воронка продаж (лид → договор) | §1 | Лиды, контакты, воронки, статусы | Лиды, контакты, примечания, задачи | Нейропродажник, n8n |
| Генерация и модификация КП | §2 | Данные лида/контакта для КП | Примечания с КП, обновление статуса | Нейропродажник, proposal-generator |
| Создание договора | §3 | Данные покупателя, сделки | Документ договора, примечание, задача | Proposal-generator, менеджер |
| Онлайн-аналитика | §4 | — | Обновление меток лида | sales-analytic |
| Офлайн-аналитика | §5 | Лиды, сделки, конверсии | — | sales-analytic |
| Настройка API (OAuth2) | §6 | — | — | Общее |
| Эффективность рекламы | §7 | Лиды, сделки | — | sales-analytic |
| Деанонимизация | §8 | — | Создание лида, связь с conversation | sales-analytic / n8n |
| Обогащение KB | §9 | Примечания, диалоги менеджеров | — | Нейроаналитик |
| Chat Hub (внутренние пользователи) | §1.4 | Сделки, задачи | Создание/поиск сделок, задач | Менеджер через n8n |

Итого:
- **Чтение** amoCRM ограничений нет: лиды, контакты, сделки, воронки, статусы, примечания, каталоги (если есть).
- **Запись** нужна для: лиды, контакты, примечания, задачи, обновление статусов. Для отладки и тестов нужны тестовые объекты с явной пометкой и правилами удаления.

---

## 2. Два уровня интеграции: n8n и FastAPI

### 2.1. n8n (как сейчас)

- Один раз настроенный OAuth2 и токены в n8n.
- Workflow вызывают amoCRM (HTTP Request и т.п.), оркестрируют сценарии (создание лида → примечание → задача).
- Подходит для: вебхуки с сайта/Telegram, Chat Hub, сложные цепочки.

### 2.2. FastAPI (независимый слой)

- Отдельный сервис или модуль (например, `amocrm-service` или модуль внутри `sales-analytic` / `sales-agent`), который по `.env` (те же `AMOCRM_*`, что и для n8n) ходит в amoCRM API напрямую.
- Зачем:
  - вызовы из кода без прохождения через n8n (скрипты, фоновые джобы, аналитика);
  - единый клиент с обновлением токена (refresh) и повторными запросами при 401;
  - возможность в будущем вынести токены в отдельный сервис (если понадобится).

Токен, который уже работает в n8n, можно использовать и в FastAPI: достаточно иметь в `.env` те же `AMOCRM_SUBDOMAIN`, `AMOCRM_ACCESS_TOKEN`, `AMOCRM_REFRESH_TOKEN`, `AMOCRM_CLIENT_ID`, `AMOCRM_CLIENT_SECRET`, `AMOCRM_REDIRECT_URI` и реализовать обновление токена по 401 (как в [AMOCRM_API_SETUP.md](./AMOCRM_API_SETUP.md)).

## 3. Нейропродажник и нейроаналитик

### 3.1. Нейропродажник

- Чтение: лид, контакт, история сделок (для персонализации).
- Запись: создание/обновление лида, примечания (резюме диалога, КП), задачи при handoff.
- Может идти через n8n (webhook → workflow) или напрямую через FastAPI (если сервис сам дергает amoCRM).

### 3.2. Нейроаналитик

- Чтение: лиды, сделки, воронки, примечания — для отчётов, конверсий, эффективности рекламы (§5, §7).
- Важный кейс: **анализ диалогов менеджера с клиентом**:
  - источник: звонки/встречи → **транскрибация** (внешний сервис или свой пайплайн);
  - отбор **успешных** диалогов (сделка закрыта, высокий NPS и т.п.);
  - извлечение паттернов (вопросы, возражения, удачные ответы);
  - **пополнение раздела «промпты» в KB** (таблица `knowledge_base`, категория/метаданные вроде `sales_script` / `objection_handling`, см. [KB_ENRICHMENT_SOURCES.md](./KB_ENRICHMENT_SOURCES.md)).

Таким образом, нейроаналитик использует amoCRM для чтения (сделки, примечания) и для контекста «успешности», а результат (тексты для промптов) попадает в PostgreSQL (KB), а не в amoCRM.

---

## 4. Договор и коммерческое предложение (КП)

- **КП**: генерируются в нашей системе (таблица `proposals` в PostgreSQL), синхронизируются в amoCRM в виде примечаний к сделке (как в §2 BUSINESS_PROCESSES).
- **Договор**: создаётся на основе финализированного КП (§3), данные покупателя читаются из amoCRM; документ сохраняется у нас и при необходимости прикрепляется/дублируется в amoCRM (примечание или файл).
- Каталога товаров в amoCRM может не быть — тогда позиции КП/договора формируются из наших данных (см. ниже).

---

## 5. Каталог товаров: PostgreSQL при отсутствии списка в amoCRM

В amoCRM может не быть каталога товаров или он пустой. В [db-init/05_create_products.sql](../db-init/05_create_products.sql) уже есть таблица `products` с опциональной связью `amocrm_catalog_id`. Предлагается:

- **Источники данных для `products`**: сайт заказчика (парсинг/импорт), консультации с заказчиком, ручное заполнение. При наличии каталога в amoCRM — синхронизация в обе стороны или только чтение из amoCRM.
- **КП и договоры**: формировать состав (позиции) из `products` и при необходимости из других таблиц PostgreSQL; в amoCRM передавать только итог (текст КП/договора в примечании или файл), без привязки к каталогу amoCRM.
- Ограничений на таблицы в PostgreSQL нет — при необходимости можно добавить таблицы под специфичные типы товаров или справочники.

## 6. Тестовые объекты в amoCRM (лиды, воронки)

Чтобы отлаживать интеграцию без влияния на боевые данные, нужны явно помеченные тестовые объекты и правила их удаления.

### 6.1. Соглашение по тестовым объектам

- **Именование**: единый префикс, например `[TEST]` или `[BRATS-TEST]` в названии лида/сделки и при необходимости в примечании.
- **Тег**: отдельный тег в amoCRM, например `brats_test` или `test_integration`, чтобы фильтровать и удалять пачкой.
- **Документирование**: в репозитории хранить короткий файл (например, `docs/AMOCRM_TEST_OBJECTS.md`) с:
  - описанием назначения тестовых лидов/воронок;
  - списком тегов/префиксов;
  - пошаговой инструкцией «как удалить все тестовые объекты» (по тегу/по фильтру в amoCRM или через скрипт, вызывающий API удаления).

### 6.2. Создание тестовых объектов

- Предпочтительно через FastAPI или отдельный скрипт: создание лида/сделки с именем `[BRATS-TEST] ...` и тегом `brats_test`.
- При создании через n8n — те же правила (префикс + тег) и описание в документации.

### 6.3. Удаление

- Вручную: в amoCRM фильтр по тегу `brats_test` → удаление.
- Полуавтоматически: скрипт (Python) с использованием нашего amoCRM-клиента: получить список лидов по тегу → удалить через `DELETE /api/v4/leads`. Документировать в `AMOCRM_TEST_OBJECTS.md`.

## 7. Чтение из amoCRM — без ограничений

Поддерживаем чтение (через n8n и/или FastAPI) для:

- Лиды (список, по id, фильтры).
- Контакты, компании.
- Сделки (воронки, статусы), история по контакту (как в [AMOCRM_DEAL_HISTORY.md](./AMOCRM_DEAL_HISTORY.md)).
- Примечания к сделкам/контактам (для аналитики и обогащения KB).
- Задачи (для отображения менеджерам).
- Справочники: воронки, статусы, поля (custom fields) — для динамической схемы и отчётов ([AMOCRM_FIELDS_AND_ANALYTICS.md](./AMOCRM_FIELDS_AND_ANALYTICS.md)).
- Каталоги и элементы каталога — если в amoCRM они есть.

Реализация: общий клиент amoCRM с методом `_request` и автообновлением токена; отдельные методы `get_leads`, `get_lead`, `get_contacts`, `get_contact`, `get_pipelines`, `get_lead_notes` и т.д.

## 8. Запись в amoCRM — минимально необходимое + тест

- **Для продакшена**: создание/обновление лидов, контактов, примечания, задачи, обновление статуса сделки. По возможности через существующие n8n workflow; при необходимости — дублирование или перенос логики в FastAPI.
- **Для отладки**: создание только тестовых лидов (и при необходимости тестовых контактов/сделок) с префиксом и тегом по §6. Остальную запись в боевые сущности при разработке минимизировать или выполнять только через явные «тестовые» воронки.

## 9. Поток: диалоги менеджера → транскрибация → обогащение KB (промпты)

Это сценарий для нейроаналитика.

1. **Источник**: записи разговоров менеджера с клиентом (звонки, встречи). Хранятся у заказчика или передаются в нашу систему (файлы/ссылки).
2. **Транскрибация**: речевой текст (свой сервис или внешний API). Результат — текст диалога с разметкой «менеджер / клиент» (и при необходимости временными метками).
3. **Привязка к сделке**: по контакту/лиду в amoCRM связать транскрипт со сделкой; взять статус сделки и результат (закрыта успешно / неуспешно).
4. **Отбор успешных диалогов**: критерии — сделка в «успешно реализовано», длительность диалога, отсутствие эскалаций и т.д.
5. **Извлечение паттернов**: с помощью LLM выделить вопросы клиента, возражения, удачные ответы менеджера, ключевые фразы закрытия.
6. **Пополнение KB**: добавить в таблицу `knowledge_base` записи с:
   - `metadata.category` = `sales_script` или `objection_handling`;
   - `metadata.source` = `manager_dialogue` или `transcription`;
   - при необходимости `metadata.context.use_case` = `objection` / `closing` и т.д.  
   Так формируется раздел «промпты» для RAG нейропродажника.

Реализация может быть: скрипт (batch) или эндпоинт в FastAPI (загрузка транскрипта + amocrm_lead_id → анализ → запись в KB). Транскрибация может выноситься в отдельный сервис/воркфлоу.

## 10. Рекомендуемые шаги реализации

1. **Документация тестовых объектов**  
   Создать `docs/AMOCRM_TEST_OBJECTS.md` с соглашениями (§6) и инструкцией по удалению.

2. **Клиент amoCRM в Python**  
   Модуль с OAuth2 refresh и методами для чтения (лиды, контакты, сделки, примечания, воронки, статусы). Использовать переменные из `.env`, как в [AMOCRM_API_SETUP.md](./AMOCRM_API_SETUP.md).

3. **FastAPI-слой**  
   Либо отдельный микросервис `amocrm-service`, либо роуты в существующем сервисе (например, `sales-analytic` или общий API-шлюз). Эндпоинты только на чтение в первой итерации; опционально — создание тестового лида с тегом/префиксом.

4. **Создание тестовых лидов**  
   Скрипт или POST-эндпоинт: создание лида с именем `[BRATS-TEST] ...` и тегом, описанным в `AMOCRM_TEST_OBJECTS.md`.

5. **Продукты и КП**  
   Оставить формирование состава КП/договора из PostgreSQL (`products` + при необходимости другие таблицы); синхронизация с amoCRM — только итоговый текст/файл в примечании.

6. **Обогащение KB из диалогов**  
   Отдельный скрипт или эндпоинт: вход — транскрипт + идентификатор сделки/лида в amoCRM; логика отбора успешных диалогов и добавления записей в `knowledge_base` с категорией для промптов.

После этого интеграция с amoCRM будет поддерживать и n8n, и независимый FastAPI в соответствии с описанными бизнес-процессами, с явными правилами для тестовых данных и пополнением раздела промптов в KB из успешных диалогов менеджеров.

## 11. Реализованные артефакты

- **[amocrm-api/](../amocrm-api/)** — отдельная директория для разработки и тестирования интеграции с amoCRM (FastAPI-сервис и клиент):
  - `client.py` — клиент с автообновлением токена, чтение лидов/контактов/воронок/примечаний/каталогов, создание и удаление лидов;
  - `api/main.py` — эндпоинты GET для чтения и POST `/api/test-leads` для создания тестового лида.
- Соглашения по тестовым объектам и удалению — в [AMOCRM.md](./AMOCRM.md) (раздел 4).
- **[docs/AMOCRM.md](./AMOCRM.md)** — единый документ: описание API amocrm-api, примеры использования, результаты обследования и выводы о бизнес-процессах пользователя.
