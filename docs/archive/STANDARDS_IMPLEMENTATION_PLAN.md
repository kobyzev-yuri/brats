# План внедрения стандартов: PostgreSQL + тестовая воронка AmoCRM

Компактный план на основе анализа AmoCRM (см. [AMOCRM.md](./AMOCRM.md)): диалоги и документы не стандартизированы, разбросаны в примечаниях; каталогов и календарей в AmoCRM нет. Цель — завести **источник правды** в PostgreSQL (календарь показов, цены, шаблоны КП/договоров) и в AmoCRM использовать **одну тестовую (неиспользуемую) воронку** для тестовых лидов/контактов с минимизацией помех персоналу и постепенным переходом заказчика на стандарты.

**Ограничения:** объём изменений компактный; опция «такси до объекта» откладывается.

---

## 1. Что уже есть

| Компонент | Где | Примечание |
|-----------|-----|------------|
| Цены/объекты | PostgreSQL `products` | Уже есть (05_create_products.sql); каталога в AmoCRM нет — КП/договоры из products |
| КП | PostgreSQL `proposals` | Уже есть (06_create_proposals.sql); связь с conversation, amocrm_lead_id |
| Диалоги | PostgreSQL `conversations` + `messages` | Есть (03_create_conversations.sql); amocrm_lead_id, slots (телефон, имя и т.д.) |
| Тестовые лиды AmoCRM | amocrm-api, [BRATS-TEST] | POST /api/test-leads, тег brats_test; воронка не зафиксирована в коде |

---

## 2. PostgreSQL: что добавить (компактно)

### 2.1 Календарь показов

Одна таблица **`viewing_slots`** — слоты для записи на просмотр.

- **Поля (минимум):** id, settlement_id (посёлок), object_id (FK на products или текст «объект N»), slot_start (timestamp), slot_end (timestamp), status ('free' | 'booked' | 'completed' | 'cancelled'), contact_phone, contact_name, amocrm_lead_id (после создания лида), conversation_id (откуда записали), created_at, updated_at.
- **Идея:** n8n/бот при «хочу записаться на просмотр» читает свободные слоты из этой таблицы, предлагает время; при подтверждении — обновляет слот (status=booked, contact_*), создаёт/обновляет лид в AmoCRM (тестовая воронка) и при желании примечание «Запись на показ: дата, время, объект».

### 2.2 Шаблоны КП и договоров

Одна таблица **`document_templates`** (или две маленькие — `proposal_templates`, `contract_templates` — по желанию).

- **Поля (минимум):** id, type ('proposal' | 'contract'), name (название шаблона), body (text или jsonb для структурированного шаблона), settlement_id (опционально), is_active, created_at, updated_at.
- **Идея:** генерация КП/договора по шаблону + подстановка данных из лида/conversation/products; в AmoCRM по-прежнему только примечание/файл и смена статуса, а «канон» шаблона — в PostgreSQL.

Конкретный выбор: одна таблица `document_templates` с полем `type` — проще и компактнее.

---

## 3. AmoCRM: тестовая воронка, минимум помех

- **Выбор воронки:** использовать одну **неиспользуемую** воронку для всех тестовых лидов (например «Квалификация» 10112058, если её не используют в бою, или отдельная «Тест нейропродажник» при возможности). Зафиксировать в конфиге/документации: `AMOCRM_TEST_PIPELINE_ID`, `AMOCRM_TEST_STATUS_ID` (стартовый статус).
- **Правила:** все лиды от чата/n8n создавать с префиксом `[BRATS-TEST]` и тегом `brats_test` в эту воронку; контакты создавать/привязывать по телефону/имени; примечания — кратко, структурировано (например «Источник: сайт. Запись на показ: дата, время. Телефон: …»).
- **Цель:** персонал видит тестовые лиды в одной воронке, может игнорировать или переносить в боевую по готовности; со временем заказчик может ввести такие же стандарты в рабочих воронках.

## 4. Последовательность шагов (компактно)

1. **Миграция PostgreSQL**
   - Добавить таблицу `viewing_slots` (календарь показов).
   - Добавить таблицу `document_templates` (type, name, body, is_active) для шаблонов КП и договоров.
   - Файл: `db-init/07_viewing_slots_and_templates.sql` (таблицы `viewing_slots`, `document_templates`). Запускать после 01–06.
- Образцы данных: `db-init/08_seed_samples.sql` — 2 объекта в `products` (Инноваторы Клуб: Black Box 140 м², White Box 120 м²), 2 шаблона в `document_templates` (КП, договор), до 12 слотов в `viewing_slots` на ближайшие 5 дней. Повторный запуск не дублирует шаблоны и слоты.

2. **Конфиг AmoCRM**
   - В `config.env.example` и документации: `AMOCRM_TEST_PIPELINE_ID`, `AMOCRM_TEST_STATUS_ID` (опционально).
   - В amocrm-api при создании тестового лида — использовать эту воронку (и статус), если заданы.

3. **n8n / бот**
   - При «запись на просмотр» + контакт: (a) записать слот в `viewing_slots` (или обновить), (b) создать/обновить лид (и контакт) в AmoCRM в тестовую воронку, (c) примечание с датой/временем/объектом.
   - Диалоги по желанию продолжать писать в PostgreSQL (conversations/messages) и при появлении лида — проставлять amocrm_lead_id.

4. **Шаблоны**
   - Заполнить `document_templates` одним-двумя шаблонами КП (и при необходимости договора) для генерации из products/conversation; интеграция «генерация КП по шаблону → примечание в AmoCRM» — отдельный небольшой шаг после стабилизации показов.

5. **Такси**
   - Не реализуем в текущем объёме; в промпте/KB можно оставить формулировку «при необходимости обсудим организацию поездки», без автоматизации.

---

## 5. Сводка по стандартам

| Что | Где хранится | AmoCRM |
|-----|--------------|--------|
| Календарь показов | PostgreSQL `viewing_slots` | Только отражение в примечании к лиду (дата/время/объект) |
| Цены, объекты | PostgreSQL `products` | Не дублируем (каталога нет) |
| Шаблоны КП/договоров | PostgreSQL `document_templates` | Генерация файла/текста → примечание к сделке |
| Диалоги | PostgreSQL `conversations` + `messages` | Резюме/ключевые факты в примечании лида при handoff |
| Лиды/контакты (тест) | AmoCRM тестовая воронка | [BRATS-TEST], тег brats_test, один pipeline_id |

Так заказчик получает единый источник правды в PostgreSQL и предсказуемую тестовую зону в AmoCRM без риска для текущих процессов.
