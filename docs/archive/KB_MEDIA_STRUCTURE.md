# Структура KB для релевантных картинок и документов

Чтобы агент (нейропродажник) мог «предъявлять» релевантные запросу картинки и документы, в базе знаний используется единая схема метаданных. Схема БД не меняется: всё хранится в поле `metadata` (JSONB) чанков.

---

## 1. Метаданные чанка

### 1.1. Изображения: `metadata.images`

Массив объектов с полями:

| Поле | Тип | Описание |
|------|-----|----------|
| `url` | string | Обязательно. URL изображения (с сайта, Yandex.Disk, или отдаваемый вашим сервером, например `/static/...` или `/api/images/...`) |
| `description` | string | Описание (подпись для клиента) |
| `alt` | string | Альтернативный текст (доступность, превью) |

Пример:

```json
"images": [
  { "url": "https://example.com/photo.jpg", "description": "Дизайнерский ремонт, гостиная", "alt": "Гостиная" },
  { "url": "http://localhost:8001/static/uploads/plan.png", "description": "Планировка", "alt": "Планировка" }
]
```

- При импорте по URL и сохранении блоков/галереи в KB Admin эти данные уже попадают в `metadata.images`.
- Картинки с диска нужно раздавать по HTTP (статическая папка или API) и в KB хранить только URL.

### 1.2. Документы: `metadata.documents`

Массив объектов для PDF, DOCX, ссылок на файлы:

| Поле | Тип | Описание |
|------|-----|----------|
| `url` | string | Обязательно. Ссылка на документ |
| `title` | string | Название для отображения («Презентация для риелтора», «Прайс») |
| `description` | string | Краткое описание (опционально) |

Пример:

```json
"documents": [
  { "url": "https://disk.yandex.ru/.../presentation.pdf", "title": "Презентация для клиента", "description": "" },
  { "url": "http://localhost:8001/static/docs/price.pdf", "title": "Актуальный прайс" }
]
```

- При сохранении через `/api/mmkb/save` можно передать поле `documents` в `metadata` (или отдельным полем запроса — см. ниже).
- В контенте чанка можно по-прежнему упоминать те же ссылки текстом; по `metadata.documents` агент и фронт однозначно понимают, что это «документы к ответу».

---

## 2. Поиск и ответ агента

- **POST /api/kb/search** возвращает для каждого результата полный `metadata`, в том числе `images` и `documents`.
- В n8n в ноде **Format KB Context**:
  - из `kb_response.results` собирается текстовый контекст для LLM;
  - из тех же результатов собираются все релевантные медиа (images + documents) и добавляются в контекст блоком «Релевантные медиа»;
  - собранные массивы передаются дальше по цепочке.
- В ноде **Prepare Response** к ответу клиенту добавляется объект `media`: `{ images: [...], documents: [...] }` (до N картинок и M документов по релевантности).
- Промпт агента предписывает: если в контексте есть «Релевантные медиа», упомянуть в ответе, что к сообщению приложены фото/документы, и кратко описать их; длинные URL в текст не вставлять.
- Чат на сайте по полю `response.media` отображает под ответом блок с картинками и ссылками на документы.

---

## 3. Импорт и сохранение в KB

- **Загрузка по URL** (и библиотекарь): изображения страницы попадают в блоки и при сохранении в KB — в `metadata.images`.
- **Ручное добавление в KB Admin**: при редактировании чанка можно задать «Ссылка на картинку (URL)» и галереи — они сохраняются в `metadata.images`.
- **Документы**: при сохранении через API или админку нужно заполнять `metadata.documents` (или поле `documents` в теле запроса к `/api/mmkb/save`, которое маппится в `metadata.documents`). Так агент сможет возвращать релевантные документы в ответе.

---

## 4. Итог

| Что | Где хранится | Кто использует |
|-----|--------------|-----------------|
| Картинки | `metadata.images` | Поиск → n8n → ответ `media.images` → чат (превью/галерея) |
| Документы | `metadata.documents` | Поиск → n8n → ответ `media.documents` → чат (ссылки «Скачать») |
| Текст | `content` + часть в `metadata` | Контекст для LLM, ответ агента |

Структура KB и промпты к агенту/библиотекарю настроены так, чтобы один и тот же механизм (поиск по тексту → возврат метаданных → формирование `media`) позволял «предъявлять» и картинки, и документы, релевантные запросу.
