# Поля amoCRM и их использование в нейроаналитике / KB

Эти заметки описывают, **как работать с полями amoCRM**, чтобы:
- не ломать существующую отчётность заказчика;
- дать максимум пользы нейроаналитику и нейропродажнику;
- использовать динамически меняющуюся схему CRM в KB.

---

## 1. Базовые принципы

1. **Не хардкодить схему в коде.**  
   Структура полей amoCRM живая: заказчик может добавлять/менять поля. Все поля нужно воспринимать как данные, а не как константы.

2. **Разделять “их” и “наши” поля.**
   - *Чужие (заказчика)* — существующие поля, от которых зависят их отчёты и бизнес‑процессы.
   - *Наши (AI / нейро)* — дополнительные поля, которые добавляем для аналитики и подсказок, с явным префиксом (например, `nl_` / `ai_`).

3. **Схема поля должна жить в БД.**  
   В PostgreSQL стоит завести таблицу описаний полей amoCRM (см. ниже) и регулярно её обновлять из API через n8n.

---

## 2. Хранение схемы полей в БД

Рекомендуемая таблица:

```sql
CREATE TABLE crm_field_definitions (
    id           SERIAL PRIMARY KEY,
    source       TEXT NOT NULL,      -- 'lead', 'contact', 'company', 'customer'
    field_id     TEXT NOT NULL,      -- внутренний ID поля в amoCRM
    code         TEXT,               -- системный код (если есть)
    name         TEXT NOT NULL,      -- человекочитаемое имя
    type         TEXT NOT NULL,      -- text, numeric, select, multiselect, date, etc
    is_custom    BOOLEAN NOT NULL,
    is_required  BOOLEAN NOT NULL DEFAULT FALSE,
    sort_order   INT,
    options      JSONB,              -- варианты списков, enum, multiselect
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Как заполнять

- n8n‑флоу периодически вызывает API amoCRM:
  - `GET /api/v4/leads/custom_fields`
  - `GET /api/v4/contacts/custom_fields`
  - и т.д.
- обновляет/вставляет записи в `crm_field_definitions`:
  - по `field_id`/`source` определяем, обновлять или вставлять;
  - обновляем `name`, `type`, `options`, `is_required`, `sort_order`.

### Зачем это нужно

- не нужно править код при каждом изменении полей amoCRM;
- можно генерировать динамические подсказки и справку в KB;
- можно использовать эту схему в аналитике (какие поля реально работают, какие — “для галочки”).

---

## 3. Какие поля обязательно уважать (со стороны заказчика)

После получения доступа к amoCRM нужно **выяснить у заказчика**, какие поля:
- входят в их стандартные отчёты и дашборды;
- критичны для KPI/аналитики;
- уже используются в триггерах/автоматизациях.

Обычно это:

- **Идентификация лида**
  - Название сделки / лида
  - Ответственный менеджер
  - Воронка/статус (`pipeline_id`, `status_id`)

- **Контактные данные**
  - Телефон(ы)
  - Email(ы)
  - Канал связи (иногда как тег или отдельное поле)

- **Источник**
  - Стандартное поле “Источник лида”
  - Теги/кастомные поля источников (Avito, сайт, Яндекс.Директ, рекомендации и т.п.)

- **Интерес/продукт**
  - Объект/категория интереса (квартира, коттедж, тариф и т.д.)

Нейроаналитик и нейропродажник должны **минимум не портить** эти поля, а лучше — заполнять их корректно и последовательно.

---

## 4. Какие свои поля стоит добавить (AI/нейро)

Это поля “усилители”, которые мы можем предложить заказчику как добавочную ценность.
Их лучше заводить с префиксом (`nl_` / `ai_`), чтобы было очевидно происхождение.

Примеры:

- **Уровень интереса**
  - `nl_intent_score` — число 0–1, рассчитывается в `AnalyticsService` по событиям сайта.
  - `nl_intent_level` — enum: `cold`, `warm`, `hot` (мэппинг из `nl_intent_score`).

- **Поведенческие признаки**
  - `nl_sessions_count` — количество сессий до создания лида.
  - `nl_pages_viewed` — глубина просмотра до заявки.
  - `nl_key_events` — JSON или enum с ключевыми событиями: `form_submit`, `phone_click`, `whatsapp_click` и т.п.

- **Сегментация**
  - `nl_segment` — гипотеза о сегменте лида (семья / инвестор / юрлицо и т.д.), выводится из поведения или диалога.

- **Приоритет**
  - `nl_priority` — приоритет обработки (A/B/C или числовой), производная от `intent_score`, источника и, возможно, бюджета.

Эти поля:
- не ломают существующие процессы;
- могут использоваться и людьми (менеджеры видят теги/подсказки), и AI‑агентами для принятия решений.

## 5. Использование полей в KB

### 5.1. Генерация “живой” справки по CRM

На основе `crm_field_definitions` можно автоматически наполнять KB материалами:

- “Какие поля есть у лида и что они значат?”
- “Какие значения у поля `Источник лида` и как мы их используем?”
- “Какие поля обязательны для заполнения на этапах QUALIFYING / PROPOSAL / CONTRACT?”

Технически:
- периодический скрипт/флоу читает `crm_field_definitions`;
- генерирует markdown‑страницы в KB (или обновляет существующие), чтобы менеджеры и агенты всегда видели актуальную структуру CRM.

### 5.2. Подсказки для нейропродажника

FSM нейропродажника может получать из KB или напрямую из БД:

- список полей, которые **должны быть заполнены** до перевода лида на следующий этап;
- текстовые подсказки:
  - “на этом этапе важно узнать бюджет и сроки принятия решения”;
  - “если поле `nl_segment` пустое, задай один из вопросов: ...”.

Это делает поведение агента **привязанным к схеме CRM**, а не к жёстко зашитому набору вопросов.

## 6. Использование полей в аналитике

1. **Связь поведения и заполненности полей**
   - Аналитика может отвечать на вопросы вида:
     - “у горячих лидов чаще всего заполнены поля X, Y, Z”;
     - “какие поля менеджеры забывают заполнять, хотя они критичны для прогноза?”

2. **Связь полей и конверсии**
   - На уровне отчётов:
     - “конверсия по лидам, у которых заполнено поле `nl_segment`, выше/ниже, чем у остальных”;
     - “как влияет выбор значения в поле `Источник лида` на конверсию hot → сделка”.

3. **Обогащение KB**
   - На основе статистики по полям можно генерировать рекомендации в KB:
     - “лиды с сегментом `Инвестор` и бюджетом > X чаще закрываются при таких условиях...”

---

## 7. Следующие шаги после получения доступа к amoCRM

1. **Снять фактическую схему**
   - Выгрузить `custom_fields` по лидам/контактам/сделкам.
   - Записать в `crm_field_definitions`.

2. **Классифицировать поля**
   - Отметить:
     - какие поля обязательны для заказчика;
     - какие используются в их текущих отчётах;
     - какие практически пустые (кандидаты на переиспользование/архивацию).

3. **Согласовать с заказчиком**
   - Показать:
     - “вот ваш текущий набор полей”;
     - “вот какие поля мы предлагаем добавить (`nl_*`) и как они повлияют на аналитику/агента”.

4. **Реализовать синхронизацию**
   - n8n‑флоу для обновления `crm_field_definitions`;
   - обновление скриптов и KB на основе этой таблицы.

Так мы получим **гибкую и прозрачную систему работы с полями amoCRM**, которая:
- уважает существующие процессы заказчика;
- даёт нам пространство для экспериментов (через `nl_*` поля);
- позволяет использовать динамическую схему CRM и в нейроаналитике, и в KB, и в работе нейропродажника.
