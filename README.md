### Обзор проекта

Мультиагентная система для застройщика коттеджного посёлка, объединяющая:
- **Нейроаналитиков** — анализ поведения на сайте и эффективности рекламных каналов
- **Нейропродажника** — экспертная переписка с лидами до передачи менеджерам

**Цель**: снижение потерь дорогих лидов, повышение конверсии, создание универсального продукта для адаптации под другие отрасли.

### Основные компоненты системы

1. **Нейроаналитик** (`sales-analytic/`)
   - Анализ поведения пользователей на сайте и эффективности рекламных каналов
   - Онлайн отчёты (real-time dashboard, мониторинг воронки)
   - Оффлайн отчёты (ежедневные, еженедельные, месячные)
   - Обнаружение горячих лидов на основе поведения
   - Триггеры для инициации нейропродажника
   
   **Ключевые документы:**
   - [`sales-analytic/README.md`](./sales-analytic/README.md) — описание сервиса и быстрый старт
   - [`sales-analytic/docs/`](./sales-analytic/docs/) — подробная документация по аналитике

2. **Нейропродажник** (`sales-agent/`)
   - Экспертная переписка с лидами до передачи менеджерам
   - FSM‑состояния воронки (GREETING, QUALIFYING, OBJECTION_1/2/3, PROPOSAL, HANDOFF)
   - Векторный поиск в базе знаний (KB) для ответов на вопросы
   - Семантический поиск объектов недвижимости по запросу клиента
   - Генерация персональных коммерческих предложений
   - Ненавязчивая деанонимизация анонимных посетителей
   
   **База знаний (Knowledge Base):**
   - Хранится в PostgreSQL с pgvector для векторного поиска
   - Содержит скрипты продаж, обработку возражений, tone of voice
   - Обогащается автоматически из диалогов и примечаний менеджеров
   - Поддерживает мультитенантность (несколько поселков)
   - **Управление через KB Service** (`kb-service/`) — FastAPI сервис для CRUD операций и импорта данных
   - **Мультимодальная база знаний**: текст + изображения, структурированные разделы под домен заказчика
   - **Агент‑библиотекарь на Gemini 3** (через `proxyapi.ru`): проверка релевантности и качества материалов, поиск дубликатов, генерация аннотаций к изображениям перед сохранением в KB
   
   **Каталог объектов:**
   - Синхронизируется с каталогом товаров amoCRM (nomenklatura)
   - Расширенная версия в PostgreSQL с векторным поиском
   
   **Ключевые документы:**
   - [`docs/CONCEPT.md`](./docs/CONCEPT.md) — принципы KB, скрипты, админка, воронка, медиа в ответе
   - [`docs/KB.md`](./docs/KB.md) — база знаний: структура, хранение, использование
   - [`docs/POSTGRESQL_ARCHITECTURE_EXPLAINED.md`](./docs/POSTGRESQL_ARCHITECTURE_EXPLAINED.md) — архитектура таблиц (KB и products)
   - [`docs/CONCEPT.md`](./docs/CONCEPT.md) — концепция KB и медиа

3. **Proposal Generator** (`proposal-generator/`)
   - Генерация персональных КП на основе потребностей клиента и каталога объектов
   - Модификация КП в процессе переговоров (изменение цены, условий, объектов)
   - Версионирование КП (история изменений)
   - Финализация КП для передачи менеджеру и использования как основы договора
   - Генерация договоров купли-продажи на основе финализированного КП
   - Интеграция с sales-agent (запрос на генерацию/модификацию)

4. **Procurement Agent** (`procurement-agent/`) — *перспективная опция*
   
   > ⚠️ **Примечание**: Procurement Agent — опциональный модуль для клиентов-организаций. Реализация планируется только при наличии достаточного времени и ресурсов. Детали архитектуры и бизнес-модели описаны в отдельном документе.
   
   - FSM процесса закупки (поиск → сравнение → переговоры → заказ)
   - Адаптеры для разных протоколов (REST, cXML, EDI)
   - Интеграция с вашим продажником и внешними поставщиками
   - Аналитика закупок и история транзакций
   
   **Документация:** Детали реализации и бизнес-модель будут описаны в отдельном документе при начале разработки.

5. **Оркестратор** (`n8n/`)
   - Workflow для координации компонентов
   - Интеграции с amoCRM, Яндекс.Метрикой и др.
   - Webhook'и с сайта, Telegram/других каналов
   - Триггеры handoff к менеджерам
   
   **Ключевые документы:**
   - [`n8n/README.md`](./n8n/README.md) — установка и настройка n8n
   - [`n8n/workflows/README.md`](./n8n/workflows/README.md) — описание всех workflow
   - [`docs/AMOCRM.md`](./docs/AMOCRM.md) — интеграция с AmoCRM

6. **База данных** (`shared-db/`)
   - PostgreSQL: основные данные (диалоги, KB, каталог объектов, КП, договоры)
   - Redis: кэш и очереди
   - pgvector: векторный поиск для RAG

### Каналы коммуникации и маршрутизация

**Внешние каналы:**
- Telegram‑бот (личные чаты) и бот в группах/каналах (мониторинг триггерных сообщений)
- Avito (новые диалоги и заявки по объявлениям)
- Сайт/веб‑чат/формы (лиды с лендингов и посадочных страниц)

**Единый маршрут через n8n:**
- Все события с каналов приходят в n8n (webhook/API)
- n8n нормализует данные и связывает событие с контактом/сделкой в amoCRM
- Далее используется один из двух маршрутов:
  - **Маршрут через AI (нейропродажник)** — дефолтный для типовых лидов
  - **Прямой маршрут к менеджеру** — для VIP/сложных/особых кейсов

**Интеграция сайта:**
- JavaScript код на сайте отслеживает ввод телефона/email и отправляет события в n8n webhook
- Подробнее: [`site-integration/README.md`](./site-integration/README.md)

### Разделение данных: PostgreSQL vs amoCRM

**amoCRM** используется для лидов и сделок до продажи:
- Лиды — базовые данные (имя, телефон, email, статус воронки)
- Сделки — коммерческая информация (сумма, этап, дата, привязка к объекту через кастомное поле)
- Контакты — контактные данные клиентов
- Задачи — напоминания менеджерам, действия
- Примечания — краткие резюме от агентов

**PostgreSQL** хранит данные, которых нет в amoCRM или которые требуют сложной структуры:
- Полная история диалогов (каждое сообщение, метаданные)
- Состояния FSM (текущий шаг воронки, слоты, счётчики)
- База знаний для RAG (таблица knowledge_base с pgvector)
- Каталог объектов недвижимости (products) с ценами и характеристиками
- Варианты скидок (discount_rules) и варианты продаж (sales_options)
- Аналитические данные (события Метрики, паттерны поведения)
- Коммерческие предложения (proposals) с версионированием
- Договоры купли-продажи (contracts)

### Технический стек

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|
| **Оркестратор** | n8n | Визуальные workflow, интеграции, webhooks |
| **Аналитика** | Python 3.11+, FastAPI | Гибкость, асинхронность, ML-экосистема |
| **LLM** | GigaChat / YandexGPT / Локальные модели | Соответствие 152-ФЗ |
| **База данных** | PostgreSQL 15+ | Надёжность, JSON-поддержка |
| **Кэш/Очереди** | Redis 7+ | Производительность, pub/sub |
| **Векторный поиск** | pgvector (расширение PostgreSQL) | RAG для базы знаний, единая БД |
| **Контейнеризация** | Docker + Docker Compose | Изоляция, масштабирование |

### Соблюдение 152-ФЗ

**Требования:**
1. **Локальные/российские LLM**: GigaChat, YandexGPT или self-hosted модели
2. **Обезличивание ПД**: перед передачей в облачные сервисы (если необходимо)
3. **Хранение данных**: на серверах в РФ
4. **Прозрачность**: уведомление пользователей о боте (если требуется)
5. **DLP-политика для зарубежных LLM/VLM**: запрет на передачу ПДн, реквизитов, "сырых" CRM‑идентификаторов

**Реализация:**
- Использование российских провайдеров LLM
- Self-hosted n8n и сервисы
- Шифрование данных в покое и при передаче
- Логирование доступа к ПД
- Внедрение слоя `sanitize_for_llm()` перед любыми вызовами внешних LLM/VLM

**DLP-фильтр:**
- Реализован в: `kb-service/services/dlp_service.py`
- Использование в нейропродажнике: `sales-agent/services/dlp_integration.py`

**Подробнее:** [`docs/DLP_FOR_SALES_AGENT.md`](./docs/DLP_FOR_SALES_AGENT.md)

### Этапы разработки

**Этап 1 (0-1 месяц): Инфраструктура и интеграции**
- Развёртывание n8n (self-hosted)
- Настройка интеграции с amoCRM
- Настройка интеграции с Яндекс.Метрикой (API)
- Создание базовых workflow в n8n
- Настройка PostgreSQL и Redis

**Этап 2 (1-2 месяц): Нейроаналитик**
- Разработка онлайн-модуля аналитики (FastAPI)
- Разработка офлайн-модуля аналитики
- Интеграция с Метрикой (вебхуки + API)
- Определение паттернов поведения
- Генерация рекомендаций по каналам

**Этап 3 (1-2 месяц, параллельно): Нейропродажник и Proposal Generator**
- Настройка LLM (GigaChat/YandexGPT)
- Настройка базы знаний
- Разработка диалог-менеджера (sales-agent)
- Логика определения момента передачи менеджеру
- Интеграция с каналами (Telegram, Avito)
- Proposal Generator: генерация, модификация и финализация КП

**Этап 4 (2-3 месяц): Интеграция и тестирование**
- Объединение модулей через n8n
- Настройка триггеров и порогов
- Тестирование на пилотном объекте
- Сбор метрик и оптимизация
- Документация и обучение пользователей
