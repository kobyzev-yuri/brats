## Общее описание

**KB Admin** — это веб-интерфейс на базе Streamlit для управления базой знаний, которая используется агентом продаж для получения релевантной информации при общении с клиентами.

### Основные возможности:

- 🔍 **Семантический поиск** по базе знаний с фильтрацией
- ✏️ **Редактирование** существующих chunks (фрагментов знаний)
- 📥 **Импорт** новых данных из текста или файлов
- 📊 **Статистика** по содержимому базы знаний
- 🗑️ **Управление** chunks (активация/деактивация, удаление)

## Запуск приложения

### Предварительные требования

1. **KB Service** должен быть запущен и доступен по адресу `http://localhost:8001`
2. **PostgreSQL** с расширением `pgvector` должен быть настроен
3. **Python окружение** с установленными зависимостями

### Запуск

```bash
cd /projects/brats/kb-service
streamlit run web/kb_admin_app.py --server.port 8501
```

Приложение откроется в браузере по адресу: `http://localhost:8501`

## Разделы приложения

### Обзор и поиск

**Назначение**: Поиск и просмотр chunks в базе знаний с возможностью быстрого редактирования.

#### Поля поиска

1. **Поисковый запрос** (текстовое поле)
   - **Назначение**: Семантический поиск по содержимому chunks
   - **Как использовать**: Введите ключевые слова или фразу (например: "коттеджный посёлок", "цены", "возражения")
   - **Особенности**: 
     - Поиск работает по смыслу, а не только по точному совпадению
     - Можно использовать `*` для поиска всех chunks
     - Пустой запрос не выполнит поиск

2. **Категория (фильтр)** (выпадающий список)
   - **Назначение**: Фильтрация результатов по категории контента
   - **Варианты**:
     - `""` (пусто) — без фильтра
     - `product_info` — информация о продукте
     - `sales_script` — скрипты продаж
     - `objection_handling` — обработка возражений
     - `target_audience` — целевая аудитория
     - `tone_of_voice` — тон общения
     - `contacts` — контакты
     - `pricing` — ценообразование
     - `location` — местоположение

3. **min_similarity** (слайдер, 0.0–1.0)
   - **Назначение**: Минимальный порог схожести для результатов поиска
   - **Как использовать**: 
     - `0.0` — показать все результаты (даже с низкой релевантностью)
     - `0.3` — показать только релевантные результаты (рекомендуется)
     - `0.6+` — показать только очень релевантные результаты
   - **По умолчанию**: `0.3`

4. **Лимит результатов** (слайдер, 5–100)
   - **Назначение**: Максимальное количество результатов для отображения
   - **По умолчанию**: `20`

#### Кнопка "Искать"

- **Назначение**: Выполняет поиск с указанными параметрами
- **Результат**: Отображает список найденных chunks

#### Результаты поиска

Каждый результат отображается в **expander** (раскрывающемся блоке) с информацией:

- **Заголовок expander**: `[ID] категория (sim=схожесть)`
  - `ID` — уникальный идентификатор chunk
  - `категория` — категория контента
  - `sim` — коэффициент схожести (0.0–1.0)

- **Содержимое expander**:
  - **Категория**: категория контента
  - **Целевая аудитория**: для кого предназначен контент
  - **Приоритет**: приоритет контента
  - **Источник**: откуда был импортирован chunk
  - **Содержимое**: полный текст chunk

- **Кнопка "Открыть для редактирования #ID"**
  - **Назначение**: Открывает форму редактирования прямо на этой же странице (под результатами поиска)
  - **Действие**: После нажатия форма редактирования появляется ниже результатов поиска

#### Форма редактирования (под результатами поиска)

После нажатия "Открыть для редактирования" под результатами поиска появляется форма редактирования с теми же полями, что описаны в разделе [Редактирование chunk](#редактирование-chunk).

**Особенности**:
- Форма появляется на той же странице, не требуется переключение вкладок
- Результаты поиска остаются видимыми выше формы
- Можно закрыть редактор кнопкой "Закрыть редактор"

### Редактирование chunk

**Назначение**: Редактирование существующего chunk. Выбор только из списка записей, реально присутствующих в базе (нельзя ввести произвольный ID).

#### Как попасть на эту страницу

1. Переключитесь на вкладку **"Редактирование chunk"** в боковой панели.
2. В выпадающем списке **"Chunk для редактирования"** выберите запись (отображаются как «ID — заголовок или начало текста»).
3. Либо на странице **"Обзор и поиск"** найдите chunk и нажмите **"Открыть для редактирования #ID"** — откроется форма редактирования под результатами или переход на страницу редактирования с уже выбранным chunk.

#### Поля формы редактирования

1. **ID** (только чтение)
   - **Назначение**: Уникальный идентификатор chunk
   - **Формат**: Число

2. **Контент** (текстовое поле, высота 250px)
   - **Назначение**: Основное текстовое содержимое chunk
   - **Как использовать**: Редактируйте текст напрямую в поле
   - **Важно**: Это основной контент, который будет использоваться агентом продаж

3. **Категория** (выпадающий список)
   - **Назначение**: Классификация контента по типу
   - **Варианты**:
     - `product_info` — информация о продукте/посёлке
     - `sales_script` — скрипты продаж и диалоги
     - `objection_handling` — обработка возражений клиентов
     - `target_audience` — описание целевой аудитории
     - `tone_of_voice` — рекомендации по тону общения
     - `contacts` — контактная информация
     - `pricing` — информация о ценообразовании
     - `location` — информация о местоположении

4. **Целевая аудитория** (выпадающий список)
   - **Назначение**: Определяет, для кого предназначен контент
   - **Варианты**:
     - `end_buyer` — конечный покупатель (физическое лицо)
     - `realtor` — риелтор/агент
     - `both` — для обеих аудиторий (по умолчанию)

5. **Приоритет** (выпадающий список)
   - **Назначение**: Приоритет контента при поиске и использовании
   - **Варианты**:
     - `high` — высокий приоритет (используется в первую очередь)
     - `medium` — средний приоритет (по умолчанию)
     - `low` — низкий приоритет

6. **Активен** (чекбокс)
   - **Назначение**: Включение/отключение chunk
   - **Как использовать**: 
     - ✅ Отмечен — chunk активен и используется в поиске
     - ❌ Не отмечен — chunk деактивирован (soft delete), не используется в поиске
   - **По умолчанию**: `True` (активен)

7. **Теги (через запятую)** (текстовое поле)
   - **Назначение**: Дополнительные метки для категоризации и фильтрации
   - **Формат**: Теги через запятую (например: `важно, цены, vip`)
   - **Как использовать**: Введите теги через запятую, пробелы автоматически удаляются

8. **Источник (source)** (текстовое поле)
   - **Назначение**: Указание источника, откуда был импортирован chunk
   - **Примеры**: `"Саммари встречи.odt"`, `"manual_import"`, `"kb_info.txt"`
   - **Использование**: Помогает отслеживать происхождение данных

#### Кнопки управления

1. **Сохранить изменения** (основная кнопка)
   - **Назначение**: Сохраняет все изменения в базе данных
   - **Действие**: 
     - Отправляет обновленные данные в KB Service
     - Показывает сообщение об успешном сохранении
     - Обновляет chunk в базе данных

2. **Закрыть редактор**
   - **Назначение**: Закрывает форму редактирования без сохранения изменений
   - **Действие**: Сбрасывает выбор chunk для редактирования

3. **Деактивировать (soft delete)**
   - **Назначение**: Деактивирует chunk без физического удаления
   - **Действие**: 
     - Устанавливает `is_active = False`
     - Chunk перестает участвовать в поиске
     - Данные остаются в базе для возможного восстановления

4. **Удалить навсегда**
   - **Назначение**: Физически удаляет chunk из базы данных
   - **Действие**: 
     - Полностью удаляет запись из таблицы `knowledge_base`
     - **Внимание**: Операция необратима!

### Блоки оператора

**Назначение**: Единое место для наполнения базы: ручные блоки (текст, картинка, ссылка на картинку), импорт документов (Word, PDF, TXT) и быстрый импорт по URL. У каждой записи всегда задаётся **источник (source)** для последующего поиска.

#### 1. Ручные блоки

- **Тип блока**: «Описание (текст)», «Картинка (загрузить файл)» или «Ссылка на картинку (URL)».
- **Заголовок**, **раздел**, **теги** — обязательны по смыслу.
- **Источник** задаётся автоматически:
  - текст — `operator_block`;
  - картинка (файл) — имя загруженного файла;
  - ссылка на картинку — URL изображения.

#### 2. Импорт документа в KB (Word, PDF, TXT)

- Загрузка файла `.txt`, `.pdf`, `.docx` или `.pptx` (презентация). Текст извлекается (из слайдов PPTX — по порядку), разбивается на chunks и сохраняется в KB.
- **Источник**: по умолчанию — имя файла (можно изменить). Если не указан URL, источником служит имя файла.

#### 3. Быстрый импорт по URL (черновик)

- Ввод URL страницы сайта заказчика, опционально раздел и теги.
- Кнопка **«Импортировать по URL через библиотекаря»** — загрузка страницы, анализ библиотекарём, смысловые блоки и галерея изображений.
- **Источник** для всех сохранённых из этого URL блоков — сам URL (для поиска по источнику).

### Статистика

**Назначение**: Просмотр статистики по содержимому базы знаний.

#### Общая статистика

Отображаются три метрики:

1. **Всего chunks**
   - **Назначение**: Общее количество chunks в базе (включая деактивированные)

2. **Активных chunks**
   - **Назначение**: Количество активных chunks (используемых в поиске)

3. **Категорий**
   - **Назначение**: Количество различных категорий в базе

#### Детальная статистика

1. **По категориям** (JSON)
   - **Назначение**: Распределение chunks по категориям
   - **Формат**: `{"category": count, ...}`
   - **Пример**: `{"sales_script": 11, "product_info": 6, ...}`

2. **По целевой аудитории** (JSON)
   - **Назначение**: Распределение chunks по целевой аудитории
   - **Формат**: `{"audience": count, ...}`
   - **Пример**: `{"both": 44, "end_buyer": 5, ...}`

3. **По приоритету** (JSON)
   - **Назначение**: Распределение chunks по приоритету
   - **Формат**: `{"priority": count, ...}`
   - **Пример**: `{"high": 44, "medium": 10, ...}`

### Мультимодальная KB и библиотекарь

**Назначение**: Добавление и проверка материалов для мультимодальной базы знаний (текст + изображения) с использованием агента‑библиотекаря.

#### Базовый сценарий работы

1. **Ручной ввод материала**
   - Введите:
     - **Заголовок материала**
     - **Раздел мультимодальной KB**:
       - `settlement_info` — посёлок и инфраструктура
       - `product_info` — объекты и планировки
       - `pricing_offers` — цены, акции, ипотека
       - `faq` — частые вопросы
       - `objections` — возражения и отработки
       - `legal_finance` — юридические и финансовые аспекты
     - **Основной текст** — подробное описание материала
     - **Теги** — через запятую (`посёлок, инфраструктура, цены`)
     - **URL‑ы изображений** — по одному в строке (картинки с сайта/материалов заказчика)

2. **Отправка на проверку библиотекарю**
   - Нажмите кнопку **"Отправить на проверку библиотекарю"**.
   - Приложение отправит материал на эндпоинт `POST /api/mmkb/review`.
   - Агент‑библиотекарь:
     - анализирует текст и указанные изображения;
     - проверяет релевантность и качество;
     - ищет возможные дубликаты в KB;
     - формирует краткое **abstract** и **ключевые пункты**;
     - возвращает структурированное решение.

3. **Просмотр решения библиотекаря**
   - Отображаются:
     - **decision**: `approve` / `reject` / `needs_review`;
     - **relevance_score**, **quality_score**, флаги `is_relevant` и `has_valuable_info`;
     - **Причина решения**;
     - **Abstract** и **ключевые пункты** (bullet‑список);
     - **Фильтрованный текст** (`filtered_content`) — текст без "воды";
     - **Проверка дубликатов** — блок `duplicate_check` (похожие документы, similarity);
     - **Аннотации к изображениям** (`image_annotations`), если они были сформированы.

4. **Сохранение материала в KB**
   - Выберите:
     - **KB category** (техническая категория в таблице `knowledge_base`):
       - `product_info`, `sales_script`, `objection_handling`, `target_audience`, `tone_of_voice`, `contacts`, `pricing`, `location`
     - **Целевая аудитория**: `end_buyer` / `realtor` / `both`
     - **Приоритет**: `high` / `medium` / `low`
   - Нажмите **"Сохранить материал в KB"**:
     - вызывается `POST /api/mmkb/save`;
     - в таблицу `knowledge_base` добавляется новый chunk:
       - `content` — основной текст;
       - `metadata` — включает:
         - `title`, `section`, `tags`, `images[]`;
         - объект `librarian` с полным решением библиотекаря.

5. **Быстрый импорт по URL**
   - В блоке "Быстрый импорт по URL (черновик)":
     - укажите **URL страницы сайта заказчика**;
     - при необходимости — **раздел** и **теги**.
  - Нажмите **"Импортировать по URL через библиотекаря"**:
    - вызывается `POST /api/mmkb/import_from_url`;
    - загружается страница, извлекаются текст и изображения; библиотекарь анализирует материал и формирует **смысловые блоки** и **аннотации к картинкам** (vision: при настроенном 3dtoday — VisionAnalyzer, иначе Gemini или GPT-4o);
    - в интерфейсе отображаются: черновой документ, решение библиотекаря (approve/reject/needs_review), список смысловых блоков с возможностью редактирования и кнопками «Добавить этот блок в KB» / «Добавить галерею в KB», а также галерея изображений с аннотациями.

## Поля и атрибуты

### Сводная таблица полей

| Поле | Тип | Обязательное | Назначение | Где используется |
|------|-----|--------------|------------|------------------|
| **ID** | `int` | ✅ | Уникальный идентификатор chunk | Редактирование, Поиск |
| **content** | `str` | ✅ | Текстовое содержимое chunk | Редактирование, Импорт, Поиск |
| **category** | `enum` | ✅ | Категория контента | Редактирование, Импорт, Поиск (фильтр) |
| **target_audience** | `enum` | ✅ | Целевая аудитория | Редактирование, Импорт, Поиск (фильтр) |
| **priority** | `enum` | ✅ | Приоритет контента | Редактирование, Импорт, Поиск (фильтр) |
| **is_active** | `bool` | ✅ | Активность chunk | Редактирование |
| **tags** | `List[str]` | ❌ | Теги для категоризации | Редактирование |
| **source** | `str` | ❌ | Источник импорта | Редактирование, Импорт |
| **chunk_size** | `int` | ✅ | Размер chunk при импорте | Импорт |
| **chunk_overlap** | `int` | ✅ | Перекрытие chunks при импорте | Импорт |
| **query** | `str` | ✅ | Поисковый запрос | Поиск |
| **limit** | `int` | ✅ | Лимит результатов | Поиск |
| **min_similarity** | `float` | ✅ | Минимальная схожесть | Поиск |

### Категории контента (category)

| Значение | Описание | Когда использовать |
|----------|----------|-------------------|
| `product_info` | Информация о продукте/посёлке | Описания объектов, характеристик, преимуществ |
| `sales_script` | Скрипты продаж | Готовые фразы, диалоги, сценарии общения |
| `objection_handling` | Обработка возражений | Ответы на типичные возражения клиентов |
| `target_audience` | Целевая аудитория | Описания сегментов клиентов, их потребностей |
| `tone_of_voice` | Тон общения | Рекомендации по стилю общения с клиентами |
| `contacts` | Контакты | Контактная информация, способы связи |
| `pricing` | Ценообразование | Информация о ценах, скидках, условиях оплаты |
| `location` | Местоположение | Информация о расположении, транспортной доступности |

### Целевая аудитория (target_audience)

| Значение | Описание | Когда использовать |
|----------|----------|-------------------|
| `end_buyer` | Конечный покупатель | Контент для физических лиц, покупающих для себя |
| `realtor` | Риелтор/агент | Контент для профессиональных агентов |
| `both` | Обе аудитории | Универсальный контент (по умолчанию) |

### Приоритет (priority)

| Значение | Описание | Когда использовать |
|----------|----------|-------------------|
| `high` | Высокий | Критически важная информация, используемая в первую очередь |
| `medium` | Средний | Стандартный контент (по умолчанию) |
| `low` | Низкий | Дополнительная информация, используемая реже |

## Советы и рекомендации

### Поиск

- Используйте **семантический поиск** — не обязательно точное совпадение слов
- **min_similarity = 0.3** — хороший баланс между релевантностью и количеством результатов
- Используйте **фильтр по категории** для более точных результатов
- Для широкого поиска используйте `*` в запросе

### Редактирование

- **Сохраняйте изменения** регулярно — изменения не сохраняются автоматически
- Используйте **теги** для дополнительной категоризации
- **Источник (source)** помогает отслеживать происхождение данных
- **Деактивация** предпочтительнее удаления — данные можно восстановить

### Импорт

- **chunk_size = 3000** — хороший баланс для большинства случаев
- **chunk_overlap = 300** — помогает сохранить контекст на границах
- Для **длинных документов** используйте больший `chunk_size` (4000–5000)
- Для **коротких фраз** используйте меньший `chunk_size` (1500–2000)
- Всегда указывайте **источник** для отслеживания данных

### Категоризация

- Используйте **правильные категории** — это критично для качества поиска
- **target_audience = "both"** — для универсального контента
- **priority = "high"** — только для критически важной информации
- **Теги** — для дополнительной детализации (например: `vip, срочно, акция`)

## Устранение проблем

### KB Service не доступен

**Симптомы**: В боковой панели отображается "KB Service: unhealthy" или ошибка подключения.

**Решение**:
1. Проверьте, что KB Service запущен: `curl http://localhost:8001/health`
2. Проверьте настройки в `kb-service/config.env`
3. Убедитесь, что PostgreSQL запущен и доступен

### Поиск не возвращает результаты

**Симптомы**: Поиск выполняется, но результатов нет.

**Решение**:
1. Уменьшите **min_similarity** до `0.0` для тестирования
2. Проверьте, что в базе есть активные chunks (раздел "Статистика")
3. Попробуйте поиск с `*` для получения всех chunks

### Ошибка при сохранении изменений

**Симптомы**: При сохранении появляется ошибка.

**Решение**:
1. Проверьте, что все обязательные поля заполнены
2. Проверьте подключение к KB Service
3. Проверьте логи KB Service для деталей ошибки

### Импорт не работает

**Симптомы**: Импорт завершается с ошибкой или не создает chunks.

**Решение**:
1. Проверьте, что текст не пустой
2. Проверьте параметры импорта (chunk_size, chunk_overlap)
3. Проверьте логи KB Service
4. Убедитесь, что KB Service имеет доступ к базе данных

## Технические детали

### API Endpoints

Приложение использует следующие endpoints KB Service:

- `GET /health` — проверка состояния сервиса
- `POST /api/kb/search` — поиск в KB
- `GET /api/kb/{chunk_id}` — получение chunk
- `PUT /api/kb/{chunk_id}` — обновление chunk
- `DELETE /api/kb/{chunk_id}` — удаление chunk
- `POST /api/kb/import` — импорт данных
- `GET /api/kb/stats` — статистика
- `POST /api/mmkb/review` — прогон мультимодального материала через библиотекаря (без сохранения)
- `POST /api/mmkb/save` — сохранение материала в KB с учётом решения библиотекаря
- `POST /api/mmkb/import_from_url` — импорт чернового материала по URL с последующей проверкой библиотекарем

### Конфигурация

Настройки приложения находятся в `kb-service/config.env`:

- `KB_API_URL` — URL KB Service (по умолчанию: `http://localhost:8001`)
- `DATABASE_URL` — строка подключения к PostgreSQL
- `KB_TABLE` — имя таблицы KB (по умолчанию: `knowledge_base`)
